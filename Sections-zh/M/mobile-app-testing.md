移动应用程序测试

涉及在公开发布之前验证移动应用程序的功能，确保满足技术和业务要求。

## 有关移动应用程序测试的问题吗？

### 基础知识和重要性

#### 什么是移动应用测试？

[Mobile app testing](../M/mobile-app-testing.md) 涉及验证为手持设备设计的应用程序的功能、可用性和一致性。这是应用程序开发的关键阶段，旨在确保应用程序在到达最终用户之前满足所需的质量标准。测试包括检查应用程序在各种条件下的行为，包括不同的网络性能、设备特定功能和操作系统版本。
  **关键方面**包括评估应用程序的：

- **用户界面 (UI)**：确保应用程序的视觉元素和导航组件在不同的屏幕尺寸和分辨率下按预期运行。
  - **用户体验 (UX)**：确认应用程序提供流畅、直观的用户体验，包括手势响应和转换。
  - **功能**：验证所有功能是否按预期正常工作。
  - **性能**：测量应用程序在负载下的行为、响应能力和资源使用情况。
  - **安全**：检查可能损害用户数据或应用程序完整性的漏洞。
  - **本地化**：确保应用程序适当适应不同的语言和地区。
  测试还应该考虑应用程序的生命周期，包括安装、更新和卸载过程。鉴于移动设备、操作系统和用户条件的多样性，通常采用真机测试以及仿真器和模拟器相结合的方式来实现全面覆盖。

[Automated testing](../A/automated-testing.md) 是[mobile app testing](../M/mobile-app-testing.md) 的组成部分，可有效执行重复测试和回归测试，确保应用程序的可靠性和稳健性。

  - **本地化**：确保应用程序适当适应不同的语言和地区。

#### 为什么移动应用测试很重要？

[Mobile app testing](../M/mobile-app-testing.md) 对于确保不同设备上应用程序的**质量**、**性能**和**安全性**至关重要。随着智能手机和平板电脑的普及，应用程序必须在各种屏幕尺寸、操作系统和硬件配置上提供无缝的用户体验。测试可以识别可能对用户满意度产生负面影响的问题，例如崩溃、响应缓慢或电池耗尽，这对于保持应用程序市场的竞争优势至关重要。
  此外，移动应用程序通常可以访问敏感数据和个人信息，这使得 [security testing](../S/security-testing.md) 对于防范漏洞和防止数据泄露不可或缺。由于移动设备在不同的网络和各种条件下运行，测试还可以验证应用程序在网络速度波动以及在 Wi-Fi 和蜂窝数据之间转换时仍然可靠。
  [Automated testing](../A/automated-testing.md) 工具简化了此过程，支持跨多个设备环境进行频繁且一致的测试，这对于 [agile development](../A/agile-development.md) 周期和持续集成工作流程至关重要。通过及早发现缺陷，[automated testing](../A/automated-testing.md) 降低了修复[bugs](../B/bug.md) 的成本并加快了上市时间，同时还使人类测试人员能够专注于需要人工关注的更复杂的[test scenarios](../T/test-scenario.md)。
  从本质上讲，[mobile app testing](../M/mobile-app-testing.md) 是提供强大、用户友好且安全的应用程序体验的支柱，这对于用户保留、声誉以及最终应用程序在竞争激烈的市场中的成功至关重要。

#### 移动应用程序测试和 Web 应用程序测试之间的主要区别是什么？

[mobile app testing](../M/mobile-app-testing.md) 和 Web 应用程序测试之间的主要区别包括：

- **环境多样性**：移动应用程序必须在各种设备、操作系统和屏幕尺寸上进行测试。 Web 应用程序通常在不同的浏览器和版本上进行测试。
  - **交互方法**：移动应用程序通常使用滑动和捏合等触摸手势，这些手势通常在 Web 应用程序中不使用。
  - **安装过程**：移动应用需要通过应用商店安装和更新，需要对安装过程进行测试。 Web 应用程序可通过浏览器访问，无需安装。
  - **资源限制**：移动设备在电池寿命、处理能力和内存方面有更多限制，这可能会影响应用程序性能并需要特定测试。
  - **连接变化**：移动应用程序在旅途中使用，需要在各种网络条件和转换下进行测试，例如从 Wi-Fi 切换到蜂窝数据。
  - **外围交互**：与大多数 Web 应用程序不同，移动应用程序可以与特定于设备的功能（如相机、GPS 和传感器）进行交互。
  - **生命周期管理**：移动应用程序具有不同的生命周期，包括后台状态和中断（如来电），需要进行测试。
  - **安全问题**：移动应用程序可能会在设备上存储敏感数据，与在服务器上存储数据的 Web 应用程序相比，会引发不同的安全问题。
  - **用户界面**：移动应用程序的 UI 是为较小的屏幕设计的，必须测试响应能力和可扩展性，而 Web 应用程序是为更广泛的屏幕尺寸设计的。
  了解这些差异对于定制 [test automation](../T/test-automation.md) 策略以有效验证移动和 Web 应用程序至关重要。

- **环境多样性**：移动应用程序必须在各种设备、操作系统和屏幕尺寸上进行测试。 Web 应用程序通常在不同的浏览器和版本上进行测试。

### 移动应用测试的类型

#### 移动应用测试有哪些不同类型？

除了基础知识之外，[mobile app testing](../M/mobile-app-testing.md) 的不同类型包括：

- **[Exploratory Testing](../E/exploratory-testing.md)** ：无脚本测试以探索应用程序功能。
  - **[Security Testing](../S/security-testing.md)** ：确保应用程序免受威胁。
  - **[Localization Testing](../L/localization-testing.md)** ：检查不同区域设置下的应用程序行为。
  - **安装测试**：验证各种设备上的安装过程。
  - **中断测试**：评估应用程序对呼叫或通知等中断的响应。
  - **恢复测试**：测试应用程序承受故障和从故障中恢复的能力。
  - **[Beta Testing](../B/beta-testing.md)** ：将应用程序发布给一组用户以在现实世界中曝光。
  - **[A/B Testing](../A/a-b-testing.md)** ：比较两个版本以确定哪个版本性能更好。
  - **一致性测试**：确保应用程序遵守标准和指南。
  - **[Load Testing](../L/load-testing.md)** ：评估高用户负载下的性能。
  - **[Stress Testing](../S/stress-testing.md)** ：确定极端条件下应用程序的稳定性。
  - **[Volume Testing](../V/volume-testing.md)** ：使用大量数据检查应用程序行为。
  - **网络测试**：测试不同网络条件下的应用程序性能。
  - **认证测试**：在发布之前验证应用程序是否符合某些认证标准。
  每种类型都针对应用程序质量和用户体验的特定方面，有助于打造强大且可靠的移动应用程序。

#### 移动应用程序中的功能测试和非功能测试有什么区别？

移动应用程序中的[Functional testing](../F/functional-testing.md) 侧重于验证应用程序的功能和操作是否按预期运行。它涉及测试应用程序的用户交互、数据处理和业务逻辑，以确保其满足指定的要求。示例包括测试用户流程、表单提交和应用内购买。
  另一方面，[Non-functional testing](../N/non-functional-testing.md) 评估在各种条件下定义应用程序质量的方面，而不是与特定行为或功能直接相关。这包括性能、安全性、兼容性、可用性和可扩展性测试。它是关于应用程序的行为和执行方式，而不是它的用途。例如，检查应用程序如何处理压力、负载，或者其针对潜在违规的安全性如何。
  总之，[functional testing](../F/functional-testing.md) 回答“它有效吗？”而[non-functional testing](../N/non-functional-testing.md) 则回答“它在各种条件下都能正常、安全地工作吗？”两者对于提供强大的移动应用程序都至关重要。

#### 移动应用程序中的可用性测试是什么？

移动应用程序的 [Usability testing](../U/usability-testing.md) 侧重于通过观察真实用户尝试在应用程序内完成任务来评估**用户体验** (UX)。它的目的是识别任何可用性问题，收集定性数据，并确定参与者对产品的满意度。与其他形式的测试不同，[usability testing](../U/usability-testing.md) 本质上是**以用户为中心**。
  关键方面包括：

- **易用性**：新用户导航应用程序的直观程度如何？
  - **使用效率**：用户熟悉界面后执行任务的速度有多快？
  - **记忆力**：一段时间不使用该应用程序后，用户恢复熟练程度有多容易？
  - **错误频率和[severity](../S/severity.md)**：用户犯错误的频率，这些错误有多严重，以及他们如何从中恢复？
  - **满意度**：使用该设计的愉快程度如何？
  [Usability testing](../U/usability-testing.md) 可以在开发的任何阶段进行，从线框到最终产品。它通常涉及**任务场景**，其中要求用户完成特定操作，而观察者做笔记或记录会话。然后使用反馈来完善 UI/UX 元素。
  在移动环境中，其他考虑因素包括**屏幕尺寸**、**触摸界面**和**移动特定功能**，例如手势和设备方向。在**实际设备**上进行测试以准确评估用户体验至关重要。
  总之，移动应用程序中的[usability testing](../U/usability-testing.md) 是一种定性方法，可确保应用程序的设计符合用户期望和可用性标准，最终产生更加**用户友好**和**成功**的产品。

- **易用性**：新用户导航应用程序的直观程度如何？
  - **使用效率**：用户熟悉界面后执行任务的速度有多快？
  - **记忆力**：一段时间不使用该应用程序后，用户恢复熟练程度有多容易？
  - **错误频率和[severity](../S/severity.md)**：用户犯错误的频率如何，这些错误有多严重，以及他们如何从中恢复？
  - **满意度**：使用该设计的愉快程度如何？

#### 什么是移动应用程序中的兼容性测试？

移动应用程序中的[Compatibility testing](../C/compatibility-testing.md) 是验证应用程序在不同设备、操作系统、屏幕尺寸、分辨率和其他变量（例如网络条件或硬件传感器）上是否按预期运行的过程。它确保应用程序提供一致的用户体验，无论其使用的设备或环境如何。
  为了有效地进行[compatibility testing](../C/compatibility-testing.md)：

- **识别最流行的设备和操作系统版本**
    并将其纳入您的测试矩阵中。

- 使用组合
    **真实设备和仿真器/模拟器**
    以覆盖广泛的场景。

- 测试
    **不同的网络类型**
    （Wi-Fi、4G、3G等）和条件（低信号、高延迟）以确保网络兼容性。

- 检查
    **正确的显示和功能**
    在各种屏幕尺寸和分辨率上。

- 验证应用程序的
    **与设备特定功能集成**
    例如相机、GPS 和加速计。

- 确保应用程序兼容
    **不同的用户设置**
    ，例如语言和辅助功能选项。
  [Automated testing](../A/automated-testing.md) 工具可用于跨多个设备执行重复的兼容性测试。但是，对于自动化测试无法捕获的细微问题，[manual testing](../M/manual-testing.md) 也可能是必要的。随着新设备和操作系统更新的发布，定期确定 [compatibility testing](../C/compatibility-testing.md) 策略的优先级并进行更新至关重要。

- **Appium**：一种开源工具，用于在 iOS 和 Android 平台上自动化本机、移动 Web 和混合应用程序。
  - **Espresso**：Android 原生测试框架，提供用于编写​​ UI 测试的 API。
  - $

    ```
    ```
Espresso.onView(withId(R.id.my_view)).perform(click());

  ```
  - **XCTest/XCUITest**: Apple's test framework for UI testing of iOS apps.
  - ```swift
  XCTAssert(app.staticTexts["Welcome"].exists)

- **早期测试**：它们允许对移动应用程序进行快速和早期的测试，甚至在实际硬件可用之前。

  - **可扩展性**：模拟器和仿真器可以轻松扩展以同时模拟多个设备，这有利于负载和压力测试。
  然而，它们无法完全取代真实设备测试的需求，因为它们可能无法准确复制所有特定于设备的行为和特征，例如电池使用情况、网络状况或传感器交互。因此，它们经常与物理设备结合使用，以实现全面的[test coverage](../T/test-coverage.md)。

#### 移动应用测试中有哪些常见挑战？

[mobile app testing](../M/mobile-app-testing.md) 中的常见挑战包括：

- **设备碎片**：许多设备具有不同的屏幕尺寸、分辨率和硬件配置，因此很难确保所有设备上的应用程序行为一致。
  - **操作系统变化**：不同版本的操作系统以及制造商特定的定制增加了测试工作的复杂性。
  - **网络多样性**：测试必须考虑各种网络速度和连接问题，包括 3G、4G、5G 和 Wi-Fi。
  - **电池消耗**：确保应用程序不会过度消耗电池是移动测试中的一个独特挑战。
  - **内存使用**：移动设备的内存有限，必须测试应用程序的内存使用效率，以防止崩溃或速度减慢。
  - **中断条件**：来电、短信、通知和其他中断可能会影响应用程序行为，必须进行测试。
  - **本地化和国际化**：应用程序必须针对不同的语言和区域设置进行测试，以确保在不同市场中正常运行。
  - **用户界面和体验**：触摸屏交互和手势需要彻底的测试，以确保流畅的用户体验。
  - **安全问题**：移动应用程序经常处理敏感数据，因此 [security testing](../S/security-testing.md) 对于保护用户信息至关重要。
  - **应用程序商店批准**：满足应用程序商店的具体准则以确保应用程序被接受可能具有挑战性。
  - **[Automated Testing](../A/automated-testing.md) 限制**：并非所有场景都可以自动化，并且由于频繁的应用程序和操作系统更新，维护自动化测试可能非常耗时。
  克服这些挑战通常需要结合手动和[automated testing](../A/automated-testing.md)、强大的设备实验室或设备场，以及在整个开发生命周期中持续测试的重点。

#### 如何克服这些挑战？

克服[mobile app testing](../M/mobile-app-testing.md) 中的挑战需要采取战略方法：

- **持续集成/持续部署 (CI/CD)** ：实施 CI/CD 管道以自动化构建、测试和部署流程。 Jenkins、GitLab CI 和 CircleCI 等工具可以促进这一点。

  ```
  pipeline:
    build:
      script:
        - build_script.sh
    test:
      script:
        - test_script.sh
    deploy:
      script:
        - deploy_script.sh
  ```

  ```
  describe.parallel('Parallel Test Suite', () => {
    test('Test 1', async () => { /* ... */ });
    test('Test 2', async () => { /* ... */ });
  });
  ```
- **人工智能和机器学习**：利用人工智能驱动的工具进行测试创建、维护和分析，以识别[flaky tests](../F/flaky-test.md)并优化[test suites](../T/test-suite.md)。
  - **[Risk-based Testing](../R/risk-based-testing.md)**：根据风险和影响确定 [test cases](../T/test-case.md) 的优先级，首先关注关键功能，以使测试过程更加高效。

  - **反馈循环**：与开发团队建立强大的反馈机制，以快速解决测试过程中发现的问题。

    确保该应用程序可供各种残障人士使用。

- **利用分析和崩溃报告**
    确定需要更多测试重点的领域并确定优先级。

- **应用测试分片**
    并行运行测试，减少总体执行时间。

- **保持平衡[test pyramid](../T/test-pyramid.md)**
    ，有大量的单元测试、适量的集成测试和少量的 UI 测试。

- **保持测试独立和幂等**
    确保它们可以按任何顺序运行，并且一项测试的结果不会影响其他测试。

- **使用网络模拟和虚拟化**
    模拟不同的网络速度和条件。

- **定期更新[test cases](../T/test-case.md)**
    以反映用户行为和应用程序功能的变化。

- **执行[exploratory testing](../E/exploratory-testing.md)**
    与自动化测试一起发现脚本测试可能遗漏的问题。

#### 如何优化移动应用测试中的性能？

要优化 [mobile app testing](../M/mobile-app-testing.md) 中的性能：

- **优先考虑关键[test cases](../T/test-case.md)**
    通过识别最常见的使用场景并关注它们。

- 使用
    **高效[test automation](../T/test-automation.md)框架**
    例如针对移动环境进行优化的 Appium 或 Espresso。

- 实施
    **并行测试**
    跨不同设备和平台同时运行多个测试。

- 通过删除不必要的步骤来优化您的测试脚本
    **重用代码**
    如有可能。

- **配置您的应用程序**
    使用 Android Profiler 或 Xcode Instruments 等工具来识别性能瓶颈。

- **模拟外部依赖项**
    例如服务器或数据库，以减少测试执行时间并提高稳定性。

- **限制模拟器/模拟器的使用**
    用于性能测试；更喜欢真实设备以获得更准确的结果。

- **清理[test data](../T/test-data.md)**
    测试运行之前和之后，以防止性能随着时间的推移而下降。

- **监控系统资源**
    例如测试运行期间的 CPU、内存和网络使用情况，以确保它们不会影响性能。

- **调整 CI/CD 管道**
    在正确的阶段进行性能测试，以便及早发现问题，而不会减慢交付过程。

- 使用
    **缓存机制**
    在适当的情况下加快测试设置和拆卸。

- **定期更新您的测试工具**
    和框架从性能改进和错误修复中受益。
