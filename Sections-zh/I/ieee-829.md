IEEE 829

是软件测试文档的标准，规定了整个测试生命周期中文档的结构。

## 相关术语：

- [Testing documentation standard](../T/testing-documentation-standard.md)

## 有关 IEEE 829 的问题吗？

### 基础知识和重要性

#### IEEE 829 标准是什么？

[IEEE 829](../I/ieee-829.md) 标准，也称为 **IEEE 软件和系统测试文档标准**，为创建测试文档提供了结构化框架。该标准概述了整个测试生命周期中测试文档的形式和内容。它包括[test plans](../T/test-plan.md)、[test design specifications](../T/test-design-specification.md)、[test case](../T/test-case.md) 规范、测试程序规范、测试项目传输报告、[test logs](../T/test-log.md)、测试[incident reports](../I/incident-report.md)、测试总结报告等文档规范。
  虽然该标准本身并不能直接确保[software testing](../S/software-testing.md)的质量，但遵守它可以促进一致性、可追溯性和问责制，这对于有效的[test management](../T/test-management.md)和评估至关重要。它作为以可理解和可重复的方式记录 [test process](../T/test-process.md) 的指南，这对于复杂的项目以及当测试工作涉及多个利益相关者时特别有益。
  [IEEE 829](../I/ieee-829.md) 标准已被 **ISO/IEC/IEEE 29119** 取代，该标准旨在为[software testing](../S/software-testing.md) 提供一组更新且国际统一的标准，可在任何软件开发生命周期中使用。然而，[IEEE 829](../I/ieee-829.md) 仍然是理解测试文档标准和实践演变的参考点。
  经验丰富的[test automation](../T/test-automation.md) 工程师可能会利用[IEEE 829](../I/ieee-829.md) 的原则来构建其自动化测试文档，确保自动化测试得到良好描述、可维护并与整体[test strategy](../T/test-strategy.md) 保持一致。

#### 为什么 IEEE 829 标准在软件测试中很重要？

[IEEE 829](../I/ieee-829.md) 标准，也称为**软件测试文档标准**，在 [software testing](../S/software-testing.md) 中非常重要，因为它提供了用于记录测试过程的**框架**和**统一的指南**。这种标准化有助于团队成员和利益相关者之间的**清晰沟通**，确保每个人都了解测试目标、计划和结果。它还促进不同项目和团队之间的**一致性**，这对于涉及多个测试人员或组织时保持质量至关重要。
  通过定义特定文档及其所需内容，[IEEE 829](../I/ieee-829.md) 有助于创建测试过程的全面记录，这对于**可追溯性**、**可审核性**和**遵守监管标准至关重要。该文档对于**审查**测试过程、**分析缺陷**以及做出有关[software quality](../S/software-quality.md) 的明智决策非常宝贵。
  此外，遵守[IEEE 829](../I/ieee-829.md) 标准可以导致**改进[test coverage](../T/test-coverage.md)** 和**更有效的[test case](../T/test-case.md) 开发**，因为它鼓励采用系统的方法进行测试设计。这可以导致**更好地检测缺陷**并最终有助于开发更可靠的软件。
  总之，[IEEE 829](../I/ieee-829.md) 标准很重要，因为它增强了[software testing](../S/software-testing.md) 中的**通信**、**一致性**和**质量**，同时还提供了支持[test process](../T/test-process.md) 的**合规性**、**审查**和**改进**的结构。

#### IEEE 829 标准如何影响软件测试的质量？

[IEEE 829](../I/ieee-829.md) 标准，也称为软件和系统测试文档标准，通过提供结构化的测试文档方法，显着影响[software testing](../S/software-testing.md) 的质量。这确保了整个测试过程的一致性、完整性和可追溯性。通过遵循[IEEE 829](../I/ieee-829.md)，测试团队创建详细且标准化的文档，从而促进更好的沟通、更清晰地理解测试目标以及更有效的[test execution](../T/test-execution.md)。
  遵守该标准可以增强测试的可重复性，从而实现可靠的 [regression testing](../R/regression-testing.md) 并更轻松地进行维护。它还有助于识别 [test coverage](../T/test-coverage.md) 差距，并支持根据预定义标准评估测试结果。该标准对文档的强调有助于责任和项目审计，以及测试过程的法律辩护。
  通过提供通用语言和一组实践，[IEEE 829](../I/ieee-829.md) 改善了团队成员之间以及与利益相关者之间的协作，从而带来更高质量的软件和更高效的测试周期。它还通过确保测试的所有方面都得到规划和记录来实现更好的风险管理，从而减少缺陷渗透到生产的可能性。
  总体而言，[IEEE 829](../I/ieee-829.md) 对[software testing](../S/software-testing.md) 质量的影响植根于其对彻底规划、可追溯性和标准化的促进，这是有效和高效的[test automation](../T/test-automation.md) 工作的关键。

### 文档

#### IEEE 829 标准概述了哪些类型的文档？

[IEEE 829](../I/ieee-829.md) 标准，也称为软件测试文档标准，概述了支持[software testing](../S/software-testing.md) 结构化方法的几种类型的文档。这些文件包括：

- **[Test Plan](../T/test-plan.md)** ：指定预期测试活动的范围、方法、资源和时间表。
  - **[Test Design Specification](../T/test-design-specification.md)** ：详细说明测试用例以及一个功能或一组功能的测试方法。
  - **[Test Case](../T/test-case.md) 规范**：描述测试用例的输入、预测结果和执行条件集。
  - **测试程序规范**：概述执行测试的步骤，包括设置、环境以及如何执行测试。
  - **测试项目传输报告**：记录测试项目向测试团队的交付情况。
  - **[Test Log](../T/test-log.md)** ：有关测试执行的相关详细信息的时间顺序记录。
  - **测试 [Incident Report](../I/incident-report.md)** ：记录测试期间发生的需要进一步调查的任何事件。
  - **测试总结报告**：提供测试活动和结果的总结，包括对相应测试项目的评估。
  这些文件旨在确保测试活动得到精心策划、系统执行和完整记录，从而促进沟通并提高测试过程的有效性。

#### IEEE 829 标准如何指导测试计划的创建？

[IEEE 829](../I/ieee-829.md) 标准提供了一种通过指定应包含的**格式**和**内容**来创建[test plans](../T/test-plan.md) 的结构化方法。它概述了确保全面测试计划的必要要素，其中包括：

- **[Test plan](../T/test-plan.md) 标识符**：测试计划的唯一名称或编号。
  - **简介**：测试计划范围和目标的简要概述。
  - **测试项目**：要测试的软件组件。
  - **要测试的功能**：需要测试的功能的详细列表。
  - **无需测试的功能**：明确说明超出范围的功能。
  - **方法**：测试中使用的总体策略和技术。
  - **项目通过/失败标准**：定义通过或失败测试的标准。
  - **暂停标准和恢复要求**：指定测试何时应暂停以及什么条件允许恢复。
  - **测试可交付成果**：列出作为测试过程的一部分要交付的所有文档和工具。
  - **测试任务**：确定任务、责任方和估计工作量。
  - **环境需求**：详细说明任何特殊的硬件、软件或数据要求。
  - **职责**：为团队成员分配特定角色。
  - **人员配置和培训需求**：概述必要的人员和任何所需的培训。
  - **时间表**：提供测试活动的时间表。
  - **风险和意外事件**：识别不可预见事件的潜在风险和计划。
  - **批准**：列出必须批准该计划的个人。

#### IEEE 829 标准中测试设计规范的目的是什么？

[IEEE 829](../I/ieee-829.md) 标准中的 **[test design specification](../T/test-design-specification.md)** 概述了特定测试级别或测试类型的**测试条件**、**[test cases](../T/test-case.md)** 和 **[test coverage](../T/test-coverage.md) 项目**。它充当需要测试什么以及如何测试的蓝图，但没有详细说明执行测试的确切步骤。本文档有助于确保测试过程涵盖软件的所有相关方面，并且系统地设计测试以发现特定类型的缺陷。
  该规范包括：

- **[Test design specification](../T/test-design-specification.md) 标识符**：文档的唯一标识符。

  - **测试技术**：用于导出测试用例的方法和途径。
  - **测试标识**：每个测试用例的命名约定或标识符。
  - **功能通过/失败标准**：确定功能是否通过测试的标准。
  通过定义这些元素，[test design specification](../T/test-design-specification.md) 有助于使测试活动与项目的目标和要求保持一致，确保测试过程更加高效和有效。它充当测试基础（例如需求或设计规范）和[test cases](../T/test-case.md)之间的纽带，提供清晰的可追溯路径并促进[test coverage](../T/test-coverage.md)和风险的评估。

#### IEEE 829 标准如何定义测试用例规范？

在[IEEE 829](../I/ieee-829.md) 标准中，**[test case](../T/test-case.md) 规范**被定义为指定一组测试条件的**输入**和**[expected results](../E/expected-result.md)** 的文档。该规范源自[test design specification](../T/test-design-specification.md)，用于确保可以执行[test case](../T/test-case.md)来验证特定要求或被测系统的一部分。
  每个[test case](../T/test-case.md) 规范通常包括：

- **[Test case](../T/test-case.md) 标识符**：测试用例的唯一标识符。
  - **测试项目**：要测试的项目或功能。
  - **输入规格**：输入的详细描述，包括执行测试所需的数据和设置。
  - **输出规格**：给定输入时应出现的预期结果。
  - **执行条​​件**：执行测试之前必须满足的任何先决条件或条件。
  - **特殊程序要求**：测试执行期间必须遵循的任何特定步骤或程序。
  - **Intercase dependency**：有关此测试用例如何与其他测试用例相关的信息（如果适用）。
  [test case](../T/test-case.md) 规范的目的是对验证软件是否满足其设计要求所需的步骤提供清晰、简洁和完整的描述。它可以指导测试人员以一致且可重复的方式执行测试并记录结果。

- **[Test case](../T/test-case.md) 标识符**：测试用例的唯一标识符。
  - **测试项目**：要测试的项目或功能。
  - **输入规格**：输入的详细描述，包括执行测试所需的数据和设置。
  - **输出规格**：给定输入时应出现的预期结果。
  - **执行条​​件**：执行测试之前必须满足的任何先决条件或条件。
  - **特殊程序要求**：测试执行期间必须遵循的任何特定步骤或程序。
  - **Intercase dependency**：有关此测试用例如何与其他测试用例相关的信息（如果适用）。

#### IEEE 829 标准中测试程序规范的作用是什么？

在 [IEEE 829](../I/ieee-829.md) 标准中，**测试过程规范**详细说明了执行测试的操作顺序。它包括设置 [test environment](../T/test-environment.md) 的步骤、[test execution](../T/test-execution.md) 的顺序以及记录结果和完成测试的过程。该规范充当测试人员遵循的脚本，确保测试的一致性和可重复性。
  测试程序规范源自 **[test cases](../T/test-case.md)** 和 **[test design specifications](../T/test-design-specification.md)**。它们将 [test design specification](../T/test-design-specification.md) 中定义的测试条件和结果转换为清晰的可执行指令。这包括指定要使用的任何 **[test data](../T/test-data.md)**、**[expected results](../E/expected-result.md)** 以及 **测试后清理** 活动。
  该规范对于自动化至关重要，因为它指导脚本的开发和工具的配置。它确保自动化测试以与 [test strategy](../T/test-strategy.md) 一致的方式执行，并确保它们产生有意义的、可比较的结果。
  以下是伪代码中的测试过程的示例：

  ```
  // Test Procedure for Login Functionality
  SETUP:
    - Initialize browser and navigate to login page.
  EXECUTE:
    - Enter valid username and password.
    - Click the login button.
  VERIFY:
    - Check if the user is redirected to the dashboard.
    - Validate that a welcome message is displayed.
  TEARDOWN:
    - Log out and close the browser.
  ```通过详细说明确切的步骤，测试程序规范有助于保持测试过程的完整性，并为评估被测系统提供明确的基础。

### 比较与评估

#### IEEE 829 标准与其他软件测试标准相比如何？

**[IEEE 829](../I/ieee-829.md) 标准**，也称为**软件和系统测试文档标准**，是一组用于创建测试文档的指南。与其他[software testing](../S/software-testing.md) 标准相比，[IEEE 829](../I/ieee-829.md) 特别**以文档为中心**。它提供了一种记录测试过程的结构化方法，包括用于创建各种测试文档的特定模板和说明。
  其他标准，例如 **ISO/IEC/IEEE 29119**，提供了更广泛的范围，不仅包括文档，还包括 [software testing](../S/software-testing.md) 的流程、技术和方法。 ISO 29119 旨在为 [software testing](../S/software-testing.md) 提供一个通用框架，与 **Agile**、**DevOps** 等其他标准和监管要求兼容。
  **[ISTQB](../I/istqb.md)**（国际[Software Testing](../S/software-testing.md) 资格委员会）认证和指南侧重于[software testing](../S/software-testing.md) 专业人员所需的知识和技能。虽然[ISTQB](../I/istqb.md) 本身并不提供标准，但它为各个级别的测试人员提供了全面的知识体系（教学大纲）和能力指南。
  相比之下，[IEEE 829](../I/ieee-829.md) 对于测试过程中要产生的**特定工件**更具规定性，这既可以被视为优势，也可以被视为局限性。它在确保完整的文档方面很强，但在适应有利于轻量级文档的现代迭代开发方法方面可能不如其他标准灵活。
  总之，[IEEE 829](../I/ieee-829.md) 是一个详细的、工件驱动的标准，与 ISO 29119 等更广泛的、面向流程的标准以及 [ISTQB](../I/istqb.md) 提供的基于知识的指南形成鲜明对比。
