# 基于会话的测试 (Session-Based Testing)
[基于会话的测试 (Session-Based Testing)](#session-based-testing)

### 参见：
- [维基百科 (Wikipedia)](https://en.wikipedia.org/wiki/Session-based_testing)

## 关于基于会话的测试的常见问题？

#### 基础与重要性
- **什么是基于会话的测试 (Session-Based Testing)？**
  基于会话的测试是一种**[探索性测试](/wiki/exploratory-testing)**技术，它将测试活动组织成限时、不受干扰的“会话 (Sessions)”。每个会话专注于应用程序的特定区域，测试人员利用其技能和创造力来发现缺陷并探索软件行为。
  **测试章程 (Test Charters)** 用于引导探索，概述会话目标而无需规定具体的**[测试用例](/wiki/test-case)**。这使得测试方法具有灵活性和适应性。
  在会话期间，测试人员维护一份**会话日志 (Session Log)** 来记录其操作、观察结果和想法。该日志对于责任追溯和为未来的测试设计提供信息至关重要。
  会话通常持续 60 到 120 分钟，确保精力集中同时防止疲劳。会话结束后，测试人员通常会与利益相关者进行**总结汇报 (Debrief)**，讨论发现并调整策略。

- **为什么基于会话的测试很重要？**
  它的重要性在于将**探索性测试**置于受控且结构化的环境中。它强调**人类洞察力**和**创造力**，允许测试人员调查脚本测试可能会遗漏的复杂场景。这种方法在发现**细微缺陷**和**易用性问题**方面尤为有效。
  通过在限时会话中工作，测试人员可以**集中精力**测试特定区域，从而更深入地检查软件。**会话日志**作为详细记录，为测试过程提供了**透明度**和**责任追溯**，这对持续改进极为宝贵。此外，它非常适合**敏捷环境**，能够快速反馈并适应变化。

- **基于会话的测试与其他测试方法有何不同？**
  区别主要在于其**探索性**和对非脚本测试的**结构化方法**。与依赖预定义**测试用例**的传统脚本测试不同，它允许测试人员实时**探索**并根据软件行为做出**反应**。
  与**自动化测试**相比，它需要测试人员的**积极参与**和决策。与**随机测试 (Ad-hoc Testing)** 相比，它更具**组织性**，包含**限时会话**、**特定目标**和**章程引导**。**会话日志**增强了责任追溯和可追溯性。

- **会话的关键组成部分有哪些？**
  - **章程 (Charter)**：关于会话目标或目的的陈述，引导探索。
  - **时间盒 (Time Box)**：预定义的时长（通常 60-120 分钟）。
  - **测试人员 (Tester)**：执行测试的个人。
  - **会话日志 (Session Log)**：记录操作、观察和发现。
  - **总结汇报 (Debrief)**：会话后的讨论和发现审查。
  测试过程中会结合探索性测试、场景测试、笔记记录和证据抓取（截图、日志、视频）。

- **会话日志在其中的作用是什么？**
  会话日志是测试过程的实时记录。其作用包括：
  - **文档化**：记录执行过程、设计决策及推理。
  - **证据**：作为测试覆盖范围的证据。
  - **沟通**：确保护送至开发和利益相关者的发现具有透明度。
  - **分析**：通过发现的缺陷、耗时和覆盖范围分析会话有效性。
  - **反馈**：为后续会话改进提供基础。
  对于自动化工程师，会话日志可以与**[自动化测试](/wiki/automated-testing)**结果关联，提供全面的测试全景视图。

#### 实施
- **会话如何结构化？**
  典型的会话结构包括：
  1. 定义明确的**测试章程**。
  2. 携带测试想法进行**会话规划**。
  3. 在会话范围内进行**探索性测试**。
  4. 实时在**会话日志**中记录发现。
  5. 进行**总结汇报**以审查分析结果。
  6. **汇报洞察**以辅助决策。

- **开展基于会话的测试有哪些步骤？**
  1. **明确任务**。
  2. **创建章程**。
  3. **环境准备**。
  4. **开始会话**。
  5. **记录笔记**（包含 **[Bug](/wiki/bug)**、异常点）。
  6. **保持航向**（避免偏离目标）。
  7. **总结汇报**。
  8. **报告结果**。
  9. **审查与适应**。
  10. **回顾 (Retrospective)**。

- **常用工具有哪些？**
  常用工具聚焦于**追踪**、**记录**和**报告**：
  - **Rapid Reporter**：轻量级实时记录工具。
  - **TestRail**：支持文档化探索性测试。
  - **[JIRA](/wiki/jira)**：通过 Zephyr 或 Xray 插件进行管理。
  - **Trello**：使用看板组织会话。
  - **SessionWeb**、**qTest**、**Testpad**。

- **如何将其集成到现有流程中？**
  - 在新功能或复杂交互中**识别价值区域**。
  - **分配时间**，使其与自动化测试互补。
  - 定义与自动化策略一致的**章程**。
  - 利用**自动化准备环境**。
  - 将会话日志集成到测试管理工具中。
  - 定期审查结果并**优化自动化脚本**（覆盖发现的新场景）。

#### 收益与挑战
- **基于会话的测试有哪些收益？**
  **集中测试**、**探索性方法**、**责任追溯**、**快速反馈**、**灵活性**、**高效率**、**协作性**以及提供质量**度量指标**。

- **有哪些挑战或局限性？**
  **可扩展性**（管理大量会话）、**一致性**（不同人结果差异）、**文档压力**、**量化难度**、**对个人技能的依赖**以及**与自动化的集成难度**。

- **如何衡量其有效性？**
  - **测试覆盖率**分析。
  - **缺陷发现数**（尤其是高**[优先级](/wiki/priority)**缺陷）。
  - **会话时长**与产出比。
  - **测试人员反馈**。
  - **漏测率**（发布后发现的未捕获缺陷）。
  - **章程完成度**。

#### 现实应用
- **现实应用案例：**
  - **游戏行业**：模拟真实用户路径发现意外行为。
  - **Web 应用**：测试电商网站动态购买流程。
  - **[移动应用测试](/wiki/mobile-app-testing)**：探索不同设备和系统下的交互。
  - **网络安全**：渗透测试人员模拟攻击者行为并记录。
  - **用户体验 (UX) 测试**：观察并记录用户旅程中的困惑。

- **在敏捷开发中如何应用？**
  它与**[敏捷开发](/wiki/agile-development)**原则一致，支持**协作**、**灵活性**和**快速迭代**。通常与 Sprint 同步，用于发现脚本遗漏的缺陷。它能快速调整策略，并在每日站会或回顾会中分享见解，提升团队沟通。

- **对提高软件质量有何帮助？**
  它通过结构化框架下的专注探索，能识别可用性问题和边缘案例，填补自动化测试的空白。**总结汇报**形成的反馈闭环能不断精进测试方法。与自动化结合，实现了更全面的**[测试覆盖率](/wiki/test-coverage)**。
