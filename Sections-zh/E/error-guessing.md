## 关于错误猜测的问题？

#### 基础与重要性
- **软件测试中的错误猜测是什么？**
**错误猜测 (Error Guessing)** 是一种测试用例设计技术，测试人员根据其**直觉、经验和对系统的理解**，在没有任何正式方法论的情况下预测 Bug 可能发生的地方。它依赖于测试人员对常见陷阱和特定系统怪异行为的了解。
在实践中，它涉及基于“预感”或过去处理类似应用的经验来创建测试。

- **为什么错误猜测很重要？**
    - **补充系统化方法**：它填补了结构化测试可能遗漏的空白，针对非传统或边缘场景。
    - **识别独特的 Bug**：结构化测试基于规范，而错误猜测是动态的，能发现那些未在文档中定义的、出乎意料的问题。
    - **作为安全网**：提供一层“人为洞察”，使测试过程更稳健、更全面。

- **错误猜测与其他技术的区别？**
    - **缺乏正式结构**：与边界值分析或等价类划分等基于模型的方法不同，它更多是**随机 (Ad-hoc)** 和**启发式 (Heuristic)** 的。
    - **高度主观**：它依赖于个人的技能和经验，难以在团队间完全复制或无损转移。
    - **灵活性**：它不需要大量的文档准备，可以快速针对可疑区域发起测试。

#### 优势与劣势
- **优势**：
    - **直观**：利用直觉快速定位弱点。
    - **灵活**：适应不同上下文，无需繁琐程序。
    - **高效**：快速定位易错区域，节省时间。
- **劣势**：
    - **主观性强**：结果随测试人员的经验水平波动。
    - **不可靠的覆盖度**：可能无法完全覆盖所有错误场景，尤其是在复杂系统中。
    - **不可重复性**：缺乏正式方法，难以在团队中大规模复制。

#### 策略与实施
- **常用策略**：
    - **基于经验的边界测试**：猜测输入范围边缘的错误。
    - **压力点预测**：预测在高负载或极端条件下的崩溃点。
    - **资源耗尽预测**：猜测在内存或磁盘不足时的行为。
    - **外部系统交互**：猜测与 API 或数据库交互时的失败。
    - **并发问题**：寻找竞态条件 (Race Conditions) 或死锁。

- **如何提升技能**：
    - **回顾缺陷历史**：分析过去的 Bug 及其规律。
    - **深入了解领域**：学习软件所在的业务领域。
    - **建立检查清单 (Checklist)**：将积累的常见错误类型模型化。
    - **像普通用户一样思考**：模拟真实或极端的终端用户操作。

- **在自动化测试中的应用**：
可以将错误猜测转化为具体的**自动化脚本**或断言。例如，猜测某个输入框对特殊字符处理不周，则编写数据驱动的脚本专门注入这些字符。

#### 成效评估
可以通过以下指标衡量：
- **缺陷检测率 (DDR)**：错误猜测发现的 Bug 占总 Bug 的比例。
- **缺陷检测效率 (DDE)**：在错误猜测上花费的时间与其产出的 Bug 比例。
- **缺陷严重程度**：它是否捕捉到了常规测试漏掉的关键性问题。
