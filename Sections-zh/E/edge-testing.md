（又名边界测试）

边缘测试

，经常与“

边界测试

，”是一种测试技术，用于识别极端操作参数下可能发生的问题，通常称为软件功能或限制的“边缘”。它侧重于测试系统在其容量限制或操作极限或接近其性能或行为的情况下的性能或行为。例如，如果软件声称支持最多 1,000 个并发用户，

边缘测试

将涉及对接近 1,000 位用户（如果不是精确的话）的系统进行测试，以观察其行为。目标是确保系统在其边界可靠运行，并发现仅在极端条件下出现的潜在问题。

## 有关边缘测试的问题吗？

### 基础知识和重要性

#### 软件测试中的边缘测试是什么？

[Edge testing](../E/edge-testing.md) 专注于软件应用程序的极端边界，确保系统在其极限下正确运行。它针对最外层参数，并与其他测试类型（例如边界值分析）相结合，以验证应用程序的稳健性。
  **主要好处**包括识别软件功能外围的潜在崩溃或意外行为，否则这些行为可能会被忽视。这对于防止可能导致生产中系统故障的边缘情况相关缺陷至关重要。
  实施[edge testing](../E/edge-testing.md) 涉及识别系统的限制，例如最大容量或边界条件，并精心设计[test cases](../T/test-case.md) 将软件推向这些极端。常见技术包括使用最大值、最小值、零以及略高于或低于边界值进行测试。
  [edge testing](../E/edge-testing.md) 的工具范围从传统的[test automation](../T/test-automation.md) 框架到专门的[boundary testing](../B/boundary-testing.md) 工具。识别边缘需要彻底了解系统的规范，并且通常需要开发人员和领域专家的输入。
  [edge testing](../E/edge-testing.md) 中的挑战可能包括难以识别所有边缘情况以及可能存在大量[test scenarios](../T/test-scenario.md)。通过根据风险对边缘情况进行优先级排序并使用 [equivalence partitioning](../E/equivalence-partitioning.md) 等技术来减少 [test case](../T/test-case.md) 计数，可以缓解这些问题。
  在敏捷中，[edge testing](../E/edge-testing.md) 是迭代集成的，在每个冲刺中识别和测试边缘情况。 [edge testing](../E/edge-testing.md) 的自动化是通过在自动化框架中编写 [test cases](../T/test-case.md) 脚本来实现的，可以作为持续集成管道的一部分来触发。
  **有效[edge testing](../E/edge-testing.md) 的最佳实践**包括根据风险确定优先级、尽可能实现自动化以及与开发团队保持密切合作以了解系统的边界。

#### 为什么边缘测试在软件开发中很重要？

[Edge testing](../E/edge-testing.md) 在软件开发中至关重要，因为它通过关注边界条件和极端场景来确保应用程序的稳健性和可靠性。通过针对功能输入范围的外围，[edge testing](../E/edge-testing.md) 揭示了传统测试方法可能会遗漏的问题，而传统测试方法往往集中于典型的[use cases](../U/use-case.md)。这种形式的测试对于验证软件在最大压力或最小资源下的行为方式以及确保它优雅地处理意外或极端输入特别重要。
  在实践中，[edge testing](../E/edge-testing.md) 涉及：

- **确定关键边界**
    通过要求和规范。

- **设计[test cases](../T/test-case.md)**
    将软件推向极限。

- **执行这些测试**
    观察系统行为。
  可以使用 [Selenium](../S/selenium.md) 或 JUnit 等工具将自动化 [edge testing](../E/edge-testing.md) 集成到 CI/CD 管道中，并使用专门设计的脚本来挑战应用程序的边界。这种自动化允许在整个开发生命周期中持续验证边缘情况。
  忽视[edge testing](../E/edge-testing.md)可能会导致软件脆弱并且在野外容易发生故障，而现实世界的使用通常包括意外和极端的条件。通过整合[edge testing](../E/edge-testing.md)，开发人员可以先发制人地解决潜在的故障点，从而产生更具弹性和用户友好的应用程序。

- **确定关键边界**

- **设计[test cases](../T/test-case.md)**

#### 边缘测试与其他类型的测试有何不同？

[Edge testing](../E/edge-testing.md) 与其他类型的测试不同，它特别关注软件的**边界条件**，而这些在其他测试方法中经常被忽视。 [unit testing](../U/unit-testing.md)、[integration testing](../I/integration-testing.md) 和[system testing](../S/system-testing.md) 在典型条件下验证软件的功能和性能，而[edge testing](../E/edge-testing.md) 则针对更可能发生缺陷的**极端操作能力**和**极端情况**。
  与一般的 [functional testing](../F/functional-testing.md) 不同，[edge testing](../E/edge-testing.md) 会验证软件在正常输入下是否按预期工作，[edge testing](../E/edge-testing.md) 会故意通过**意外输入**或**极端操作场景**将软件推向极限。这包括使用最大值、最小值、零或刚好超出预期输入范围的测试。它还涉及测试软件对异常或意外用户行为的响应。
  [Edge testing](../E/edge-testing.md) 的独特之处在于它**预测失败**而不是确认功能。它需要一种旨在打破系统的心态，以找出其弱点。这一点至关重要，因为它可以发现在将软件部署到现实环境中之前可能不会出现的问题，在现实环境中软件会受到各种用户行为和条件的影响。
  在实践中，[edge testing](../E/edge-testing.md)经常涉及**自动化[test scripts](../T/test-script.md)**，它可以快速、重复地测试边界条件。这些脚本的测试方法更加激进，对软件的错误处理和稳健性提出了挑战。
  通过关注边缘，[test automation](../T/test-automation.md) 工程师不仅可以确保软件在理想情况下正常运行，而且在遇到意外情况时也**有弹性且可靠**。

#### 边缘测试的主要优点是什么？

**[edge testing](../E/edge-testing.md)** 的主要优点包括：

- **增强可靠性**：通过关注边界条件，边缘测试可确保应用程序在其极限下正确运行，从而形成更强大、更可靠的软件产品。
  - **提高质量**：边缘情况通常会揭示通过传统测试方法可能无法发现的隐藏错误。解决这些边缘情况可以显着提高软件的整体质量。
  - **风险缓解**：测试输入和输出范围的极端值有助于防止生产中潜在的系统故障，从而降低代价高昂的停机或数据丢失的风险。
  - **用户满意度**：边缘测试有助于确保应用程序可以处理意外的用户行为，从而带来更好的用户体验并提高用户满意度。
  - **合规性和安全性**：对于受监管的行业或安全关键型应用，边缘测试对于确保符合标准和维护安全至关重要。

#### 不执行边缘测试的潜在后果是什么？

跳过 [edge testing](../E/edge-testing.md) 可能会导致一些负面结果：

- **缺陷风险增加**
    在生产中，特别是在应用程序经常失败的边界条件下。

- **用户体验差**
    ，因为用户在与处于极限的系统交互时可能会遇到意外的行为。

- **安全漏洞**
    可能会被忽视，因为攻击者经常利用边缘情况。

- **系统崩溃**
    或者在边缘条件下可能会发生未处理的异常，从而导致数据丢失或服务可用性。

- **集成问题**
    如果边缘情况影响软件与其他系统或组件的交互方式，则可能会出现这种情况。

- **法律和合规风险**
    如果软件无法处理法规或标准要求的边缘情况。

- **维护成本较高**
    由于需要修复在开发周期早期未发现的问题。

### 实施和技术

#### 软件测试过程中如何实施边缘测试？

[Edge testing](../E/edge-testing.md) 是通过一系列战略步骤在 [software testing](../S/software-testing.md) 流程中实现的：

1. **识别关键路径**
    以及应用程序中最容易受到边缘情况影响的功能。

2. **定义边界**
    对于每个函数或字段，例如最大、最小和非标称输入值。

3. **设计[test cases](../T/test-case.md)**
    专门针对这些边界，包括边缘处、边缘内部和边缘外部的值。

4. **自动化[test cases](../T/test-case.md)**
    使用与应用程序的技术堆栈相符的合适框架或工具。

5. **集成边缘测试**
    进入持续集成/持续部署（CI/CD）管道，以确保它们定期执行。

6. **分析测试结果**
    识别与预期行为的任何偏差并解决潜在问题。

7. **完善测试**
    基于代码更改和新发现的边缘情况，保持测试套件的相关性和有效性。

  ```
  // Example of an automated edge test case in TypeScript
  import { expect } from 'chai';
  describe('Edge Testing Example', () => {
    it('should handle maximum input value', () => {

      const result = processInput(maxInput);
      expect(result).to.be.a('string').and.satisfy(someCondition);
    });
  });

#### 边缘测试中有哪些常用技术？

**[edge testing](../E/edge-testing.md)** 中使用的常用技术包括：

- **边界值分析 (BVA)**：在输入范围的边界进行测试。例如，如果输入字段接受 1 到 10 之间的值，请使用值 1、10 以及 0 和 11 等无效值进行测试。
  - **[Equivalence Partitioning](../E/equivalence-partitioning.md)**：将输入数据划分为可以应用[test cases](../T/test-case.md) 的等效分区。这可确保每个分区至少测试一次。
  - **[Error Guessing](../E/error-guessing.md)**：利用经验猜测最有可能发生错误的区域，包括边缘情况。
  - **[Decision Table Testing](../D/decision-table-testing.md)**：创建规则表来确定基于不同输入组合的操作，这有助于识别边缘条件。
  - **[State Transition Testing](../S/state-transition-testing.md)**：检查满足某些条件或事件触发从一种状态到另一种状态的转换时有限状态机的行为。
  - **成对测试（全对测试）**：使用所有可能的离散参数组合进行测试，以确保覆盖各种边缘情况。
  - **组合测试**：与成对类似，但涉及组合两个以上参数来测试交互和边缘情况。
  - **基于模型的测试**：使用模型来模拟可能出现边缘条件的现实场景，然后基于这些模型开发[test cases](../T/test-case.md)。
  - **[Exploratory Testing](../E/exploratory-testing.md)**：在没有预定义[test cases](../T/test-case.md) 的情况下积极探索软件，通过临时测试和学习来查找边缘情况。
  自动化这些技术通常涉及使用针对边缘条件的参数编写[test cases](../T/test-case.md) 脚本并将其集成到[test suite](../T/test-suite.md) 中。 [Selenium](../S/selenium.md)、JUnit 或 TestNG 等工具可用于自动执行边界值和[equivalence partitioning](../E/equivalence-partitioning.md) 测试，而组合或基于模型的测试可能需要更复杂的工具或框架。

#### 边缘测试常用哪些工具？

**[edge testing](../E/edge-testing.md)** 的常用工具包括：

- **[Selenium](../S/selenium.md) [WebDriver](../W/webdriver.md)** ：自动化 Web 浏览器，允许您在不同的 Web 应用程序上测试边缘情况。

#### 如何识别软件应用程序中要测试的“边缘”？

- **审查要求**：仔细检查功能规范和用户故事，以了解预期输入和行为的限制。
  - **分析数据**：查看输入字段的数据类型和范围。考虑最大值、最小值和差一值。
  - **模型用户行为**：像最终用户一样思考，识别异常或极端的用例，包括误用或意外的操作序列。
  - **探索接口**：检查 API 端点、文件上传和其他接口的限制和意外输入行为。
  - **使用[Equivalence Partitioning](../E/equivalence-partitioning.md)** ：将输入数据分解为逻辑组，其中测试用例可以应用于每个分区。
  - **边界值分析**：专门针对等价分区的边界。
  - **[Error Guessing](../E/error-guessing.md)** ：利用经验猜测应用程序可能失败的潜在弱点。
  - **风险分析**：根据潜在影响和边缘故障的可能性确定测试的优先级。
  - **自动化**：编写自动化测试，系统地改变输入数据以覆盖边缘条件。

  }

### 挑战和解决方案

#### 边缘测试期间可能会遇到哪些挑战？

[edge testing](../E/edge-testing.md) 中的挑战通常源于边缘情况的**复杂性**和**不可预测性**。以下是一些具体的挑战：

- **识别边缘情况**：很难预见所有可能的边缘情况，尤其是在复杂的系统中。
  - **再现性**：边缘情况可能很难一致地再现，这会使测试过程变得复杂。
  - **环境[Setup](../S/setup.md)**：创建发生边缘情况的确切条件可能需要复杂的设置和配置。
  - **时间限制**：考虑到边缘情况的罕见性，投入时间来测试它们可能被认为是低效的，特别是在紧迫的期限内。
  - **资源分配**：边缘测试可能需要额外的资源，例如专用工具或硬件，而这些资源可能不容易获得。
  - **间歇性问题**：一些边缘情况会导致间歇性故障，这些故障非常难以跟踪和修复。
  - **[Test Coverage](../T/test-coverage.md)** ：对于非常大或复杂的应用程序来说，实现包括所有潜在边缘情况的高测试覆盖率几乎是不可能的。
  - **自动化复杂性**：由于其独特且通常复杂的性质，自动化边缘情况测试可能具有挑战性。

#### 边缘测试如何集成到敏捷开发流程中？

  在每个冲刺期间，与其他测试活动一起优先考虑边缘情况。使用**行为驱动开发 ([BDD](../B/bdd.md))** 框架来定义接受标准并考虑到边缘情况。这允许以所有团队成员都能理解的语言编写自动化[test scripts](../T/test-script.md)。
  利用**持续集成 (CI)** 管道定期执行边缘测试。将支持 [edge testing](../E/edge-testing.md) 的工具集成到 CI 环境中，以在构建过程中运行这些测试。这可确保立即反馈代码更改对边缘情况的影响。
  合并**结对编程**或**群体测试会话**，开发人员和测试人员在其中协作探索潜在的边缘情况并确保自动化测试覆盖它们。这促进了对[edge testing](../E/edge-testing.md)重要性的共同理解，并促进了质量的集体所有权。
  Utilize **[test-driven development](../T/test-driven-development.md) (TDD)** to write edge case tests before the actual code is implemented.这确保了代码的开发从一开始就考虑到了边缘情况。
  最后，在冲刺回顾期间，审查 [edge testing](../E/edge-testing.md) 实践的有效性并根据需要进行调整。这种持续改进将完善[edge testing](../E/edge-testing.md) 的方法，使其成为[Agile development](../A/agile-development.md) 流程的一个组成部分。

- **Create detailed [test cases](../T/test-case.md)**
    对于每个已识别的边缘场景。确保它们清晰简洁，并提供精确定义如何重现边缘条件的步骤。

- **利用参数化测试**
    有效地覆盖边界处的一系列输入值。这种方法允许您为不同的数据集重用测试逻辑。

- **合并[negative testing](../N/negative-testing.md)**
    确保系统正常处理无效或意外的输入。

- **自动化边缘情况测试**
    尽可能提高覆盖范围和效率。使用支持数据驱动测试的工具来自动化多个场景。

- **定期审查和更新边缘案例**
    随着应用程序的发展。系统中的新功能或变化可能会引入新的边缘条件。

- **与开发人员合作**
    了解系统的局限性并设计与应用程序架构相关的边缘案例。

- **记录调查结果并分享知识**
    整个团队提高对潜在边缘相关问题的认识并培育质量文化。
