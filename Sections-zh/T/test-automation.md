测试自动化

涉及使用工具来运行测试并将实际结果与

预期结果

。这些工具可以简化手动流程或与持续集成系统集成。

## 有关测试自动化的问题吗？

### 基础知识和重要性

#### 什么是测试自动化？

[Test automation](../T/test-automation.md) 是使用专门软件来控制测试执行并将实际结果与预测结果进行比较的做法。它可以在已经到位的正式测试流程中自动执行重复但必要的任务，或者执行难以手动完成的额外测试。 [Test automation](../T/test-automation.md) 对于持续集成和持续交付 (CI/CD) 管道至关重要，可在软件开发和部署期间实现频繁且可靠的测试。
  自动化脚本一旦创建，就可以一遍又一遍地运行，无需额外成本，并且比手动测试快得多。它们可以模拟多个用户与网络、软件和 Web 应用程序交互。
  [Test automation](../T/test-automation.md) 通常涉及自动化 [dynamic testing](../D/dynamic-testing.md) 任务，包括：

- **单元测试**：测试软件的各个单元或组件。
  - **集成测试**：确保应用程序使用的不同模块或服务能够很好地协同工作。
  - **功能测试**：测试产品的功能和操作行为，以确保它们符合规格。
  - **回归测试**：确保先前开发和测试的软件在更改后仍然可以运行。
  - **负载测试**：检查系统是否可以处理指定的数据负载和用户交互。

  ```
  // Example of a simple automated test script in JavaScript using a testing framework
  describe('Homepage', function() {
    it('should load successfully', function() {
      browser.url('https://example.com');
      expect(browser.getTitle()).toBe('Example Domain');
    });
  });
  ```自动化需要在学习和设置工具方面进行初始投资，但如果有效实施，它可以显着提高测试效率和准确性。

- **单元测试**：测试软件的各个单元或组件。

#### 为什么测试自动化很重要？

[Test automation](../T/test-automation.md) 至关重要，因为它**增强了[software testing](../S/software-testing.md) 的效率、有效性和覆盖范围**。通过自动化重复且耗时的任务，团队可以专注于更复杂的测试场景，并确保以**一致**和**可靠**的方式对应用程序进行彻底测试。自动化支持**持续集成和交付** (CI/CD) 管道，从而能够频繁且及早地检测缺陷，从而通过在开发生命周期中更早地解决这些问题来降低修复 [bugs](../B/bug.md) 的成本。
  此外，它允许[test cases](../T/test-case.md)的**并行执行**，显着减少测试周期所需的时间。这在敏捷和 DevOps 环境中尤其重要，在这些环境中，速度和频繁发布是常态。 [Test automation](../T/test-automation.md) 还通过允许执行大量手动执行可能不切实际的测试来改进 **[test coverage](../T/test-coverage.md)**。
  此外，自动化测试可以同时在多个平台和设备上运行，确保软件在各种环境下都能正常运行。这对于验证应用程序的**跨平台兼容性**至关重要。
  最后，[test automation](../T/test-automation.md) 提供**可追踪、可重复和可审计**结果，有助于提高[software quality](../S/software-quality.md) 并符合行业标准。它生成详细的日志和报告，有助于快速识别问题并就正在测试的软件的质量做出明智的决策。

  - **可靠性**：自动化测试每次运行时都会精确地执行相同的步骤，从而消除人为错误。

#### 测试自动化有哪些缺点？

尽管有很多好处，[test automation](../T/test-automation.md) 也有一些缺点：

- **初始投资**：建立测试自动化环境需要在工具、基础设施和培训方面进行大量的前期投资。
  - **维护开销**：自动化测试可能很脆弱，需要定期维护才能跟上应用程序的变化，这可能非常耗时。
  - **学习曲线**：团队可能需要学习新的语言或框架，这可能会延迟测试自动化的初始实施。
  - **复杂性**：为复杂场景创建强大且可靠的自动化测试可能具有挑战性，并且可能仍然需要手动干预。
  - **[False Negatives](../F/false-negative.md) 和正面**：自动化测试可能会产生漏报（应该通过的测试却失败）和误报（应该失败的测试却通过），这可能会导致对自动化过程的不信任。
  - **覆盖范围有限**：自动化只能测试其编程要测试的内容，并且可能会错过人类测试人员可能发现的意外问题。
  - **工具限制**：测试自动化工具可能存在限制，无法与某些元素或应用程序交互，从而导致测试覆盖范围出现差距。
  - **环境差异**：测试可能在受控测试环境中通过，但由于测试脚本中未考虑到差异而在生产中失败。
  总而言之，虽然[test automation](../T/test-automation.md)可以极大地提高测试效率和覆盖范围，但它并非没有挑战和局限性。它需要仔细的规划、持续的维护以及与[manual testing](../M/manual-testing.md)的平衡，以确保全面的[quality assurance](../Q/quality-assurance.md)。

#### 手动测试和自动化测试有什么区别？

[Manual testing](../M/manual-testing.md) 涉及人类测试人员在没有工具或脚本帮助的情况下执行[test cases](../T/test-case.md)。测试人员在应用程序中手动执行操作以验证功能、观察结果并记录结果。这种方法本质上较慢且更容易出现人为错误，但允许直观和[exploratory testing](../E/exploratory-testing.md)，这可以发现自动化测试可能遗漏的问题。
  另一方面，[Automated testing](../A/automated-testing.md) 使用脚本和软件工具自动运行测试、管理[test data](../T/test-data.md) 并评估结果。它对于重复和回归任务更快、更可靠，因为它消除了人为错误，并且可以批量或连续运行测试。但是，它需要初始 [setup](../S/setup.md) 和维护 [test scripts](../T/test-script.md)，这可能很复杂且耗时。
  主要区别是：

- **执行**：手动测试由人类执行；自动化测试由机器执行。
  - **速度**：设置后，自动化测试速度明显加快。
  - **准确性**：自动化测试更加一致，并且不易出现人为错误。
  - **成本**：手动测试需要较少的前期投资，但从长远来看，由于执行速度较慢且需要更多资源，成本可能会更高。
  - **复杂性**：自动化测试可以处理复杂的测试场景，但需要技术技能来编写这些测试的脚本。
  - **灵活性**：手动测试更能适应变化，并且可以更好地解释细微的行为。
  - **[Setup](../S/setup.md) 时间**：自动化测试需要时间来设置和开发测试脚本。
  在实践中，平衡的方法通常会产生最佳结果，自动处理大量回归和重复任务，而[manual testing](../M/manual-testing.md) 则专注于探索性、可用性和临时测试场景。

- **执行**：手动测试由人类执行；自动化测试由机器执行。
  - **速度**：设置后，自动化测试速度明显加快。

### 工具和技术

#### 测试自动化有哪些流行的工具？

流行的[test automation](../T/test-automation.md) 工具包含各种旨在满足不同测试需求的框架和系统。这是一个简洁的列表：

- **[Cypress](../C/cypress.md)** ：基于 JavaScript 的端到端测试框架，在浏览器中运行，提供实时重新加载和交互式调试。
  - **JUnit**：Java 的单元测试框架，广泛用于测试驱动开发。
  - **TestNG** ：与 JUnit 类似，但具有更高级的功能，如注释、参数化测试和对数据驱动测试的支持。
  - **Appium**：一种开源工具，用于自动化 iOS 和 Android 上的移动应用程序以及 Windows 桌面应用程序。
  - **Espresso**：Android 移动测试框架，提供一组丰富的 API 来编写 UI 测试。
  - **XCTest** ：Apple 的 iOS 应用程序测试框架，与 Xcode 集成。
  - **机器人框架**：用于验收测试和验收测试驱动开发（ATDD）的关键字驱动测试自动化框架。
  - **Cucumber**：支持行为驱动开发（BDD），允许用简单的语言规范应用程序功能和行为。
  - **[Postman](../P/postman.md)** ：API 测试工具，允许您为 RESTful API 创建和运行自动化测试。
  - **SoapUI** ：用于测试 SOAP 和 REST API 的工具，为服务模拟和模拟提供全面支持。
  - **HP UFT（以前称为 QTP）**：一种用于功能和回归测试的商业工具，具有可视化界面和脚本支持。
  - **Katalon Studio**：与 Selenium 和 Appium 集成的综合工具，为创建自动化测试提供用户友好的界面。

#### 这些工具之间有什么区别？

- **[Selenium](../S/selenium.md)**：支持多种浏览器和语言的开源工具。它主要用于 Web 应用程序测试，需要编码技能才能创建 [test scripts](../T/test-script.md)。

#### Selenium 是什么以及它如何在测试自动化中使用？

[Selenium](../S/selenium.md) 是一个开源 **[test automation](../T/test-automation.md) 框架**，主要用于自动化 Web 应用程序。它支持多种浏览器，例如 Chrome、Firefox、IE 和 Edge，并允许使用各种编程语言（包括 Java、C#、Python、Ruby 和 JavaScript）编写测试。
  [Selenium](../S/selenium.md)的核心组件包括：

- **[Selenium](../S/selenium.md) [WebDriver](../W/webdriver.md)** ：直接与浏览器交互，绕过基于 JavaScript 的自动化需求，并允许更复杂的场景和交互。
  - **[Selenium](../S/selenium.md) Grid** ：允许在不同的机器和浏览器上同时运行测试，这有助于加快测试套件的执行速度并允许跨浏览器测试。
  - **[Selenium IDE](../S/selenium-ide.md)** ：一个浏览器扩展，用于记录用户与浏览器的交互并回放它们以自动进行测试。它对于创建快速错误重现脚本和探索性测试非常有用。
  [Selenium](../S/selenium.md) 在[test automation](../T/test-automation.md) 中使用，方法是编写脚本来指示浏览器执行哪些操作，例如单击按钮、输入文本和导航到页面，然后断言预期结果以验证 Web 应用程序的功能。这些脚本可以集成到 CI/CD 管道中以进行持续测试。
  Python 中的简单 [Selenium](../S/selenium.md) [WebDriver](../W/webdriver.md) 脚本示例：

  ```
  from selenium import webdriver
  driver = webdriver.Chrome()
  driver.get("http://www.example.com")
  element = driver.find_element_by_id("some-id")
  element.click()
  assert "Expected Text" in driver.page_source
  driver.quit()
  ```[Selenium](../S/selenium.md) 的灵活性以及与各种浏览器和操作系统的兼容性使其成为自动化基于浏览器的测试的首选。

#### 人工智能在测试自动化中的作用是什么？

人工智能通过提高 [automated testing](../A/automated-testing.md) 流程的效率、准确性和范围，在 [test automation](../T/test-automation.md) 中发挥变革性作用。它利用机器学习、自然语言处理和其他人工智能技术来改进[test automation](../T/test-automation.md)的各个方面：

- **测试生成**：人工智能可以根据用户行为、应用日志等数据源自动生成测试用例，确保全面覆盖。
  - **测试维护**：人工智能可以在应用程序 UI 或 API 发生变化时自动更新脚本，从而帮助自我修复脚本，从而减轻维护负担。
  - **不稳定检测**：人工智能算法可以识别和分析不稳定的测试，即那些产生不确定性结果的测试，并提出稳定性改进建议。
  - **视觉测试**：人工智能驱动的视觉测试工具可以比较 UI 的屏幕截图，以检测人眼通常无法注意到的像素级差异。
  - **预测分析**：人工智能可以预测关键质量指标和缺陷的可能性，使团队能够专注于高风险领域。
  - **智能诊断**：当测试失败时，人工智能可以协助进行根本原因分析，提供见解和建议以更快地解决问题。

#### 如何在项目中实现测试自动化？

要在项目中实施[test automation](../T/test-automation.md)，请执行以下步骤：

1. **评估当前的测试流程**：确定手动测试的内容以及自动化可以带来价值的地方。
  2. **定义自动化的范围**：并非所有测试都应该自动化。重点关注重复性、大容量的测试以及容易出现人为错误的测试。
  3. **选择正确的框架**：根据技术堆栈和团队的专业知识，选择符合您项目需求的框架。
  4. **设置环境**：配置必要的硬件、软件和网络设置以支持自动化工具和脚本。
  5. **开发[test scripts](../T/test-script.md)**：使用所选语言和框架编写自动化[test cases](../T/test-case.md)。确保它们是模块化的、可重复使用的且易于维护。
  6. **与 CI/CD 集成**：将 [test execution](../T/test-execution.md) 自动化，作为持续集成/持续部署管道的一部分，以获得有关代码更改的即时反馈。
  7. **审查和重构**：定期审查[test scripts](../T/test-script.md) 的相关性和准确性。根据需要进行重构以提高效率并减少不稳定。
  8. **监控和报告**：实施日志记录和报告机制来跟踪测试结果并提供有关应用程序运行状况的见解。
  9. **协作和沟通**：确保团队成员之间关于自动化流程、更新和结果的清晰沟通。
  10. **培训团队**：提供培训和资源，使团队掌握最新的自动化实践和工具。
  请记住，[test automation](../T/test-automation.md) 是一个迭代过程，需要持续评估和适应项目不断变化的需求。

#### 测试自动化的最佳实践有哪些？

[test automation](../T/test-automation.md) 的最佳实践包括：

- **优先测试**：专注于提供最大价值且手动完成时容易出现人为错误的自动化测试。
  - **保持测试独立**：每个测试应该是独立的，以避免级联失败。
  - **使用 [Page Object Model](../P/page-object-model.md) (POM)** ：此设计模式通过将页面结构与测试脚本分离来提高可维护性。
  - **实施持续集成 (CI)**：将测试自动化与 CI 管道集成，以频繁运行测试并尽早发现问题。
  - **选择正确的粒度**：单元、集成和 UI 测试之间的平衡，以确保快速反馈和全面覆盖。
  - **数据驱动测试**：外部化测试数据以轻松地使用不同的输入运行相同的测试。
  - **脚本的版本控制**：将测试代码视为生产代码；使用版本控制系统来跟踪更改。
  - **定期重构测试**：保持测试代码干净且最新，以减少维护开销。
  - **并行执行**：并行运行测试以减少执行时间。
  - **[Test environment](../T/test-environment.md)一致性** ：确保测试环境与生产环境紧密匹配，以避免错误的测试结果。
  - **监控和报告**：实施强大的日志记录和报告机制以快速识别和解决故障。
  - **处理测试不稳定**：调查并修复不稳定的测试以维持对测试套件的信任。
  - **保持更新**：跟上测试自动化工具和实践的最新趋势和更新。

#### 如何维护测试自动化脚本？

有效维护 [test automation](../T/test-automation.md) 脚本可确保它们随着时间的推移保持可靠和有价值。以下是关键做法：

- **版本控制**：使用 Git 等工具来跟踪更改，从而在需要时实现协作和回滚。
  - **模块化设计**：使用可重用的函数和对象编写模块化代码。这减少了维护并提高了可读性。

    ```
    function login(username, password) {
      // Code to perform login

#### 如何处理自动化测试中的误报？

  - **提高测试可靠性**：细化测试以使其更加稳健。这可能涉及添加显式等待、改进定位器或使用更可靠的断言方法。

### 端到端测试

#### 什么是端到端测试？

[End-to-end testing](../E/end-to-end-testing.md) 是一项**策略**，涉及验证应用程序的**集成组件**，以确保它们从开始到结束都按预期协同工作。它模拟**真实的用户场景**，有效测试系统的**外部接口**以及与外部系统的集成。此类测试对于验证**整体系统行为**至关重要，通常在**单元**和**集成测试**之后执行。
  在[end-to-end testing](../E/end-to-end-testing.md) 中，测试人员创建[test cases](../T/test-case.md)，涵盖应用程序的完整流程，例如用户交互、数据处理和后端服务。目标是识别在**实际使用**中可能发生的单元或集成测试可能会遗漏的问题，例如数据完整性、用户界面、网络通信、[database](../D/database.md) 交互和其他系统组件的问题。
  典型的端到端测试包括：

- 使用所需的数据和状态设置测试环境。
  - 通过模拟用户操作以及与应用程序的交互来执行测试。
  - 验证应用程序的行为是否符合预期，包括最终输出或状态。
  - 测试执行后清理测试环境。
  端到端测试通常是自动化的，以确保**可重复性**和**效率**，特别是在持续集成/持续部署（CI/CD）环境中。然而，由于其范围和复杂性，与其他类型的测试相比，它们的维护可能更具挑战性，并且可能需要更多的时间来执行。

- 使用所需的数据和状态设置测试环境。
  - 通过模拟用户操作以及与应用程序的交互来执行测试。
  - 验证应用程序的行为是否符合预期，包括最终输出或状态。
  - 测试执行后清理测试环境。

#### 端到端测试如何适应整体测试自动化策略？

端到端 (E2E) 测试是全面[test automation](../T/test-automation.md) 策略的关键组成部分，作为应用程序工作流程从开始到结束的最终验证。它模拟真实的用户场景，确保应用程序的所有集成组件在类似生产的环境中按预期运行。
  在典型的[test automation](../T/test-automation.md) 金字塔中，E2E 测试构成了顶端，补充了单元测试和集成测试。单元测试涵盖各个组件，集成测试验证组件之间的交互，而端到端测试则验证整个系统的行为。
  自动化端到端测试可确保复杂用户旅程的一致执行，如果手动完成，这可能会非常耗时且容易出错。优先考虑提供最大价值的 E2E 场景至关重要，因为与其他自动化测试相比，它们的维护成本更高且执行时间更长。
  E2E测试通常在临时环境中部署后执行，以确保应用程序满足业务需求。它们对于持续交付和部署 (CI/CD) 管道尤其重要，在生产发布之前充当看门人的角色。
  为了有效地将端到端测试集成到自动化策略中，重点关注：

- **选择关键用户流**

- **确保测试稳定可靠**

- **战略性地运行 E2E 测试**

#### 自动化端到端测试面临哪些挑战？

[End-to-end testing](../E/end-to-end-testing.md) 自动化面临几个挑战：

- **复杂性**：由于存在大量互连的组件和系统，自动化完整的用户流程可能会很复杂。
  - **不稳定**：由于计时问题、网络延迟或外部依赖性，测试可能会不一致地通过或失败。
  - **环境差异**：测试、登台和生产环境之间的差异可能会导致意外故障。
  - **数据管理**：设置和维护反映真实场景的测试数据很困难。
  - **UI 动态**：UI 中的更改可能会破坏测试，需要频繁更新自动化脚本。
  - **执行时间长**：端到端测试通常需要更长的时间来运行，这可能会减慢开发人员的反馈循环。
  - **资源密集型**：它们需要更多资源才能运行，因为它们通常涉及多个系统和服务。
  - **维护开销**：随着应用程序的发展，维护测试的工作量也会增加。
  - **调试困难**：在复杂的集成环境中识别故障的根本原因可能非常耗时。
  - **移动和跨浏览器问题**：确保不同浏览器和移动设备之间的一致性又增加了一层复杂性。
  为了缓解这些挑战，请考虑采取一些策略，例如优先考虑自动化的关键路径、使用可靠的定位器、实施强大的错误处理以及维护干净、结构良好的代码库。此外，与持续集成系统集成可以帮助及早发现问题。

#### 有哪些自动化端到端测试的工具？

有几种工具可以满足自动化[end-to-end testing](../E/end-to-end-testing.md)的需求，每种工具都具有适合不同测试要求的独特功能：

- **[Selenium](../S/selenium.md)**：支持多种语言和浏览器的开源框架。它具有高度可定制性，并与 TestNG 和 JUnit 等其他测试框架集成。

    ```
    WebDriver driver = new ChromeDriver();
    driver.get("http://example.com");
    ```

- **[Cypress](../C/cypress.md)**：基于 JavaScript 的工具，与应用程序在同一运行循环中运行，提供对 DOM 的本机访问。它提供了丰富的互动[test runner](../T/test-runner.md)。

    ```
    cy.visit('http://example.com');
    cy.get('.element').click();
    ```

- **Playwright**：由 Microsoft 创建，它支持使用单个 [API](../A/api.md) 跨 Chrome、Firefox 和 WebKit 进行测试。它允许在无头模式下进行测试并可以捕获屏幕截图。

    ```
    const { chromium } = require('playwright');
    (async () => {
      const browser = await chromium.launch();
      // ...
    })();
    ```

- **TestCafe**：一个[Node.js](../N/node-js.md) 工具，不需要[WebDriver](../W/webdriver.md)。它易于设置并且可以在远程设备上运行测试。

    ```
    fixture `Getting Started`
        .page `http://example.com`;
    test('My first test', async t => {
        // Test code
    });
    ```

- **Puppeteer**：一个 Node 库，提供高级 [API](../A/api.md) 通过 DevTools 协议控制 Chrome 或 Chromium。它对于测试 Chrome 扩展特别有用。

    ```
    const puppeteer = require('puppeteer');
    (async () => {
      const browser = await puppeteer.launch();
      // ...
    })();
    ```

- **Appium**：用于自动化移动应用程序的开源工具。它支持 iOS、Android 和 Windows 应用程序。

    ```
    let driver = await new Builder().forBrowser('firefox').build();
    await driver.get('http://example.com');
    ```每个工具都有其**优点**和**缺点**，选择通常取决于项目的具体需求，例如应用程序类型、[cross-browser testing](../C/cross-browser-testing.md) 所需的级别以及首选的编程语言。

- **[Selenium](../S/selenium.md)**：支持多种语言和浏览器的开源框架。它具有高度可定制性，并与 TestNG 和 JUnit 等其他测试框架集成。

    ```
    WebDriver driver = new ChromeDriver();
    driver.get("http://example.com");
    ```

- **[Cypress](../C/cypress.md)**：基于 JavaScript 的工具，与应用程序在同一运行循环中运行，提供对 DOM 的本机访问。它提供了丰富的互动[test runner](../T/test-runner.md)。

    ```
    cy.visit('http://example.com');
    cy.get('.element').click();
    ```

- **Playwright**：由 Microsoft 创建，它支持使用单个 [API](../A/api.md) 跨 Chrome、Firefox 和 WebKit 进行测试。它允许在无头模式下进行测试并可以捕获屏幕截图。

    ```
    const { chromium } = require('playwright');
    (async () => {
      const browser = await chromium.launch();
      // ...
    })();
    ```

- **TestCafe**：一个[Node.js](../N/node-js.md) 工具，不需要[WebDriver](../W/webdriver.md)。它易于设置并且可以在远程设备上运行测试。

    ```
    fixture `Getting Started`
        .page `http://example.com`;
    test('My first test', async t => {
        // Test code
    });
    ```

- **Puppeteer**：一个 Node 库，提供高级 [API](../A/api.md) 通过 DevTools 协议控制 Chrome 或 Chromium。它对于测试 Chrome 扩展特别有用。

    ```
    const puppeteer = require('puppeteer');
    (async () => {
      const browser = await puppeteer.launch();
      // ...
    })();
    ```

- **Appium**：用于自动化移动应用程序的开源工具。它支持 iOS、Android 和 Windows 应用程序。

    ```
    let driver = await new Builder().forBrowser('firefox').build();
    await driver.get('http://example.com');
    ```

#### 如何设计端到端测试的测试用例？

为[end-to-end testing](../E/end-to-end-testing.md) 设计[test cases](../T/test-case.md) 需要全面了解系统的工作流程、用户交互和集成点。这是一个简洁的指南：

1. **识别关键用户流程**：重点关注用户在应用程序中将采取的最重要路径。这些应涵盖典型的[use cases](../U/use-case.md) 和关键业务交易。
  2. **绘制场景**：创建每个用户流程的详细地图，包括替代路径和异常处理。考虑可能影响流程结果的边缘情况。
  3. **定义先决条件**：在测试开始之前建立系统的状态。这包括任何必要的数据[setup](../S/setup.md)或状态配置。
  4. **概述测试步骤**：为每个场景写出清晰简洁的步骤。包括行动、投入和预期结果。使用参数和数据驱动技术来覆盖同一流程中的变化。
  5. **检查集成点**：确保与外部系统、[databases](../D/database.md) 和服务的交互包含在[test cases](../T/test-case.md) 中，以验证整个生态系统。
  6. **包括后置条件**：定义[test execution](../T/test-execution.md)之后系统的预期状态。这可能涉及数据[verification](../V/verification.md)、系统清理或回滚步骤。
  7. **优先级和顺序**：根据[priority](../P/priority.md)、依赖性和潜在影响对[test cases](../T/test-case.md) 进行排序。这有助于优化 [test execution](../T/test-execution.md) 流程。
  8. **深思熟虑的自动化**：明智地使用自动化，同时牢记维护成本和复杂性。自动化稳定、高价值的场景，提供最佳的投资回报率。
  9. **审查和完善**：定期审查[test cases](../T/test-case.md) 的相关性和准确性，并更新它们以反映系统中的更改。
  10. **清晰记录**：确保每个[test case](../T/test-case.md)都有详细记录，使其他人易于理解和执行。为了清晰起见，请使用注释和描述性命名约定。
