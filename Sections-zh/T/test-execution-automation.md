## 关于测试执行自动化的问题？

### 基础知识和重要性

#### 什么是测试执行自动化？

[Test execution automation](../T/test-execution-automation.md)是指使用专门的软件来控制测试的执行、将实际结果与预测结果进行比较、设置测试前提条件以及其他测试控制和测试报告功能的过程。本质上，它涉及自动化手动过程，否则测试人员将逐步执行该过程。
  实际上，这意味着编写脚本或使用工具针对被测应用程序 (AUT) 运行 [test suite](../T/test-suite.md)。这些脚本与 AUT 交互，输入数据、单击按钮并提取结果以验证应用程序的行为是否符合预期。
  这是使用假设的自动化框架的伪代码的基本示例：

  ```
  testSuite("Login Feature", () => {
    testCase("Successful Login", () => {
      navigateTo("loginPage");
      enterText("usernameField", "testUser");
      enterText("passwordField", "testPass");
      click("loginButton");
      assertPage("dashboardPage");
    });
    testCase("Failed Login", () => {
      navigateTo("loginPage");
      enterText("usernameField", "testUser");
      enterText("passwordField", "wrongPass");
      click("loginButton");
      assertText("errorMessage", "Invalid credentials");
    });
  });
  ```此代码将自动执行登录功能的测试过程，包括成功登录和失败登录尝试。自动化框架通常在无头浏览器或专用[test environment](../T/test-environment.md)中执行这些测试，并报告结果以供进一步分析。

#### 为什么测试执行自动化很重要？

[Test execution automation](../T/test-execution-automation.md) 对于**扩展测试工作**和确保多次测试运行的**一致性**至关重要。它允许团队在短时间内执行大量测试，这对于[regression testing](../R/regression-testing.md) 并确保新的更改不会引入任何缺陷尤其重要。
  自动化[test execution](../T/test-execution.md)还支持**频繁集成**和**部署周期**，使团队能够有效地采用**CI/CD**实践。通过自动化执行过程，缩短了反馈循环，使开发人员能够立即了解其更改的影响。
  此外，与[manual testing](../M/manual-testing.md)相比，自动化有助于以更少的努力实现**更高的[test coverage](../T/test-coverage.md)**。它允许包含复杂的[test scenarios](../T/test-scenario.md)，由于时间或人为错误的限制，这些复杂的[test scenarios](../T/test-scenario.md)可能难以手动执行。
  在需要测试多个配置或平台的环境中，自动化可确保测试在所有所需组合中一致运行，从而**降低由于未经测试的场景而出现缺陷的风险**。
  最后，自动化[test execution](../T/test-execution.md) 为测试人员释放了宝贵的时间，使他们能够专注于更多的**战略活动**，例如[exploratory testing](../E/exploratory-testing.md) 和测试设计，而不是重复的执行任务。这种转变不仅提高了测试过程的整体质量，还提高了测试专业人​​员的工作满意度。

#### 自动化测试执行有哪些好处？

自动化 [test execution](../T/test-execution.md) 的好处包括：

- **增加[Test Coverage](../T/test-coverage.md)**：自动化允许在相同的时间内执行更多的测试，涵盖更多的功能和场景。
  - **可重复性**：测试可以以一致的精度运行多次，确保结果的可靠性。
  - **速度**：自动化测试的执行速度比手动测试快得多，显着减少了开发人员的反馈循环。
  - **效率**：创建后，自动化测试可以在不同版本的软件之间重复使用，从长远来看节省时间。
  - **成本降低**：虽然存在初始设置成本，但随着时间的推移，自动化通过最大限度地减少手动工作的需要来降低测试成本。
  - **早期[Bug](../B/bug.md) 检测**：自动化测试可以集成到构建过程中，在开发周期的早期发现问题。
  - **提高准确性**：自动化消除了人为错误的风险，从而获得更精确的测试结果。

  - **增强的报告**：可以设置自动测试来生成详细的日志和报告，提供对应用程序性能和质量的见解。

#### 测试执行自动化有哪些潜在的缺点或挑战？

[Test execution automation](../T/test-execution-automation.md) 可能会带来一些挑战：

- **初始[Setup](../S/setup.md) 成本**：建立自动化框架需要在时间和资源方面进行大量的前期投资。
  - **维护开销**：自动化测试可能很脆弱，需要定期更新才能跟上应用程序的变化。
  - **技能要求**：编写和维护脚本需要一定水平的技术专业知识。
  - **[False Positives](../F/false-positive.md)/Negatives** ：测试可能会报告并非实际错误的故障（误报）或尽管存在缺陷但仍会通过（误报）。
  - **[Test Coverage](../T/test-coverage.md)** ：实现全面的测试覆盖可能很困难；有些场景可能仍然需要手动测试。
  - **工具限制**：工具可能不支持所有技术或接口，从而导致自动化方面的差距。
  - **执行时间**：虽然自动化比手动测试更快，但复杂的测试套件可能需要相当长的时间才能执行。
  - **环境差异**：测试和生产环境之间的差异可能导致测试错误地通过或失败。
  - **数据依赖性**：自动化测试通常依赖于特定的数据集，这对于管理和保持最新可能具有挑战性。
  - **调试问题**：调查自动化测试中的失败可能非常耗时，特别是当测试未提供清晰的输出时。
  - **集成复杂性**：将测试自动化集成到 CI/CD 管道和其他系统中可能很复杂，并且需要额外的工具。
  - **可扩展性**：随着应用程序的增长，扩展测试自动化套件以匹配可能会成为一个挑战。
  应对这些挑战需要仔细规划、持续监控和定期完善自动化策略。

#### 这些工具之间有什么区别？

比较 [test automation](../T/test-automation.md) 工具时，请考虑以下主要差异：

- **语言支持**：工具因其支持的编程语言而异。例如，**[Selenium](../S/selenium.md)** 支持多种语言，如 Java、C# 和 Python，而 **[Cypress](../C/cypress.md)** 是基于 JavaScript 的。
  - **生态系统和集成**：一些工具提供更丰富的生态系统和集成。例如，**TestComplete** 提供了与其他工具的广泛集成，而 **Watir** 的开箱即用集成可能较少。
  - **易于使用**：**Katalon Studio** 等工具的设计目的是为了易于使用，并具有用户友好的界面，而 **[Selenium](../S/selenium.md) [WebDriver](../W/webdriver.md)** 等其他工具则需要更多的编码专业知识。
  - **执行速度**：执行速度可能会有很大差异。 **[Cypress](../C/cypress.md)** 和 **Playwright** 与 [Selenium](../S/selenium.md) 相比，以其快速执行时间而闻名。
  - **并行测试**：并行运行测试的能力因工具而异。 **TestNG** 与 [Selenium](../S/selenium.md) 可以并行运行测试，这在 **[Cypress](../C/cypress.md)** 中不原生支持，无需额外的插件。
  - **浏览器支持**：考虑支持的浏览器范围。 [Selenium](../S/selenium.md) 支持多种浏览器，包括较旧的浏览器，而 **Puppeteer** 主要专注于 Chrome。
  - **移动测试**：一些工具，例如**Appium**，是专门为移动测试而设计的，而其他工具则更侧重于网络。
  - **记录和回放**：**[Selenium IDE](../S/selenium-ide.md)** 等工具为测试创建提供记录和回放功能，这是无需编写代码即可快速生成测试的方法。
  - **社区和支持**：工具社区的规模和活动可能会影响您获得的支持。 [Selenium](../S/selenium.md) 拥有一个大型、活跃的社区，而较新或不太流行的工具可能只有有限的社区支持。
  - **开源与商业**：**[Selenium](../S/selenium.md)** 等开源工具可以免费使用和修改，而 **QTP/UFT** 等商业工具需要支付许可费用，但可能会提供更多专用支持和高级功能。
  - **语言支持**：工具因其支持的编程语言而异。例如，**[Selenium](../S/selenium.md)** 支持多种语言，如 Java、C# 和 Python，而 **[Cypress](../C/cypress.md)** 是基于 JavaScript 的。

  - **执行速度**：执行速度可能会有很大差异。 **[Cypress](../C/cypress.md)** 和 **Playwright** 与 [Selenium](../S/selenium.md) 相比，以其快速执行时间而闻名。
  - **并行测试**：并行运行测试的能力因工具而异。 **TestNG** 与 [Selenium](../S/selenium.md) 可以并行运行测试，这在 **[Cypress](../C/cypress.md)** 中不原生支持，无需额外的插件。
  - **浏览器支持**：考虑支持的浏览器范围。 [Selenium](../S/selenium.md) 支持多种浏览器，包括较旧的浏览器，而 **Puppeteer** 主要专注于 Chrome。
  - **移动测试**：一些工具，例如**Appium**，是专门为移动测试而设计的，而其他工具则更侧重于网络。
  - **记录和回放**：**[Selenium IDE](../S/selenium-ide.md)** 等工具为测试创建提供记录和回放功能，这是无需编写代码即可快速生成测试的方法。
  - **社区和支持**：工具社区的规模和活动可能会影响您获得的支持。 [Selenium](../S/selenium.md) 拥有一个大型、活跃的社区，而较新或不太流行的工具可能只有有限的社区支持。
  - **开源与商业**：**[Selenium](../S/selenium.md)** 等开源工具可以免费使用和修改，而 **QTP/UFT** 等商业工具需要支付许可费用，但可能会提供更多专用支持和高级功能。

#### Selenium 在测试执行自动化中的作用是什么？

[Selenium](../S/selenium.md) 在[test execution automation](../T/test-execution-automation.md) 中发挥**核心作用**，主要用于Web 应用程序。它提供了一套工具，使自动化工程师能够**与 Web 浏览器交互**并跨各种平台**自动化 Web 应用程序测试**。
  [Selenium](../S/selenium.md) [WebDriver](../W/webdriver.md) 是一个关键组件，允许创建基于浏览器的回归自动化套件和测试，从而可以使用每个浏览器对自动化的本机支持直接调用浏览器。这意味着测试的运行就像真实用户正在浏览应用程序一样，确保**高保真度**测试结果。
  使用[Selenium](../S/selenium.md)，工程师可以用多种编程语言编写[test scripts](../T/test-script.md)，包括Java、C#、Python、Ruby 和JavaScript。这种灵活性对于团队利用现有技能并与其他工具或框架集成至关重要。
  [Selenium](../S/selenium.md) 的功能包括：

- **[Cross-browser testing](../C/cross-browser-testing.md)** ：确保不同浏览器之间的兼容性。
  - **定位元素**：使用各种定位器策略识别要交互的网络元素。
  - **同步**：处理异步操作并等待元素变得可用或事件发生。
  - **[Page Object Model](../P/page-object-model.md) (POM)**：鼓励更好的测试维护并减少代码重复。
  [Selenium](../S/selenium.md) 与 TestNG、JUnit for [test management](../T/test-management.md) 等工具以及 Cucumber for [BDD](../B/bdd.md) 等框架集成。它还无缝地融入 CI/CD 管道，与 Jenkins 等工具一起工作，在构建和部署过程中实现自动化[test execution](../T/test-execution.md)。
  总之，[Selenium](../S/selenium.md) 提供了一个**强大、灵活且广泛采用的**解决方案，用于自动执行 Web 应用程序测试，使其成为 [test automation](../T/test-automation.md) 工程师工具包中的主要内容。

- **[Cross-browser testing](../C/cross-browser-testing.md)** ：确保不同浏览器之间的兼容性。
  - **定位元素**：使用各种定位器策略识别要交互的网络元素。
  - **同步**：处理异步操作并等待元素变得可用或事件发生。
  - **[Page Object Model](../P/page-object-model.md) (POM)**：鼓励更好的测试维护并减少代码重复。

#### 测试执行自动化的最佳实践是什么？

#### 如何维护和更新自动化测试脚本？

维护和更新自动化[test scripts](../T/test-script.md) 对于确保其长期有效性至关重要。以下是一些关键做法：

- **版本控制**：使用 Git 等工具来跟踪更改并管理 [test scripts](../T/test-script.md) 的版本。这样可以轻松回滚并了解随时间的变化。
  - **模块化设计**：使用可重用组件以模块化方式编写测试，以简化更新和维护。
  - **文档**：为每个[test case](../T/test-case.md)保留简洁的文档，包括目的和任何特殊注意事项，以帮助将来的更新。
  - **定期审查**：定期审查[test cases](../T/test-case.md)，以确保它们对于当前应用程序状态和要求仍然有效。
  - **重构**：持续重构测试以提高清晰度、效率和[maintainability](../M/maintainability.md)。删除已弃用的测试并更新受应用程序更改影响的测试。
  - **自动化维护任务**：在可能的情况下，自动检测 [test scripts](../T/test-script.md) 中已弃用的选择器或未使用的函数。
  - **[Test Data](../T/test-data.md) 管理**：使用可轻松更新[test data](../T/test-data.md) 的数据管理策略，例如外部数据源或数据生成工具。
  - **持续集成**：将测试维护集成到 CI/CD 管道中，以确保自动检查每个构建的测试是否存在问题。
  - **监控**：实施监控工具来跟踪一段时间内测试的性能和可靠性，提醒您潜在的维护需求。
  - **协作**：鼓励团队成员之间协作，共享知识并共同应对测试维护挑战。
  通过遵循这些实践，您可以确保您的自动化 [test scripts](../T/test-script.md) 保持稳健、可靠，并符合软件项目不断变化的需求。

- **版本控制**：使用 Git 等工具来跟踪更改并管理 [test scripts](../T/test-script.md) 的版本。这样可以轻松回滚并了解随时间的变化。
  - **模块化设计**：使用可重用组件以模块化方式编写测试，以简化更新和维护。
  - **文档**：为每个[test case](../T/test-case.md)保留简洁的文档，包括目的和任何特殊注意事项，以帮助将来的更新。
  - **定期审查**：定期审查[test cases](../T/test-case.md)，以确保它们对于当前应用程序状态和要求仍然有效。
  - **重构**：持续重构测试以提高清晰度、效率和[maintainability](../M/maintainability.md)。删除已弃用的测试并更新受应用程序更改影响的测试。
  - **自动化维护任务**：在可能的情况下，自动检测 [test scripts](../T/test-script.md) 中已弃用的选择器或未使用的函数。
  - **[Test Data](../T/test-data.md) 管理**：使用可轻松更新[test data](../T/test-data.md) 的数据管理策略，例如外部数据源或数据生成工具。
  - **持续集成**：将测试维护集成到 CI/CD 管道中，以确保自动检查每个构建的测试是否存在问题。
  - **监控**：实施监控工具来跟踪一段时间内测试的性能和可靠性，提醒您潜在的维护需求。
  - **协作**：鼓励团队成员之间协作，共享知识并共同应对测试维护挑战。

#### 如何处理测试执行自动化中的动态元素？

处理 [test automation](../T/test-automation.md) 中的动态元素需要允许您的脚本适应 UI 中的变化的策略。以下是一些技巧：

- **使用智能等待**：实现显式等待，允许您的脚本在继续之前等待某些条件（例如元素可见性）。

  ```
  WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(10));
  wait.until(ExpectedConditions.visibilityOfElementLocated(By.id("dynamicElement")));
  ```

- **动态选择器**：工艺定位器可以根据模式或关系而不是固定属性来匹配元素。

  ```
  String dynamicXpath = "//div[contains(@class, 'dynamic-class-prefix')]";
  WebElement dynamicElement = driver.findElement(By.xpath(dynamicXpath));
  ```

- **正则表达式**：在定位器中利用正则表达式将元素与动态文本或属性进行匹配。

  ```
  String regexPattern = "message_[0-9]+";
  List<WebElement> messages = driver.findElements(By.cssSelector("div[id^='message_']"))
                                    .stream()
                                    .filter(e -> e.getAttribute("id").matches(regexPattern))
                                    .collect(Collectors.toList());
  ```

- **参数化**：使用参数来处理根据用户输入或 [test data](../T/test-data.md) 更改的元素。
  - **JavaScript 执行**：执行 JavaScript 来与难以用标准 [WebDriver](../W/webdriver.md) 方法处理的元素进行交互。

  ```
  JavascriptExecutor js = (JavascriptExecutor) driver;
  WebElement element = (WebElement) js.executeScript("return document.querySelector('#dynamic');");
  ```

- **[Page Object Model](../P/page-object-model.md) (POM)** ：将与动态元素的交互封装在页面对象中以集中更改。
  请记住**保持定位器尽可能稳定**并**避免过度依赖绝对路径**。当元素高度动态时，请考虑与开发人员合作，为 UI 元素添加更稳定的标识符。

- **使用智能等待**：实现显式等待，允许您的脚本在继续之前等待某些条件（例如元素可见性）。
  - **动态选择器**：工艺定位器可以根据模式或关系而不是固定属性来匹配元素。
  - **正则表达式**：在定位器中利用正则表达式将元素与动态文本或属性进行匹配。
  - **参数化**：使用参数来处理根据用户输入或[test data](../T/test-data.md)而更改的元素。
  - **JavaScript 执行**：执行 JavaScript 来与难以用标准 [WebDriver](../W/webdriver.md) 方法处理的元素进行交互。
  - **[Page Object Model](../P/page-object-model.md) (POM)** ：将与页面对象中的动态元素的交互封装起来以集中更改。

### 集成和持续测试

#### 测试执行自动化如何适应持续集成/持续部署（CI/CD）？

[Test execution automation](../T/test-execution-automation.md) 是**CI/CD 管道**的重要组成部分，确保自动验证每个代码提交，从而尽早发现缺陷并快速向开发人员提供反馈。在 CI/CD 环境中，自动化测试通常由各种事件触发，例如新代码提交或拉取请求。
  将 [test automation](../T/test-automation.md) 集成到 CI/CD 中涉及配置 CI/CD 服务器以在成功构建后执行 [test suite](../T/test-suite.md)。这可以使用与 [test automation](../T/test-automation.md) 框架交互的插件或脚本来实现。例如，在 Jenkins 中，您可以使用以下管道脚本来运行自动化测试：

  ```
  pipeline {
      agent any
      stages {
          stage('Build') {
              steps {
                  // Build your application
              }
          }
          stage('Test') {
              steps {
                  // Run automated tests
                  sh 'npm test'
              }
          }
          stage('Deploy') {
              steps {
                  // Deploy your application if tests pass
              }
          }
      }
  }
  ```通过将自动化测试集成到 CI/CD 中，团队可以确保：

- **代码质量**
    每次提交都会维护。

- **发布**
    更可靠并且可以频繁部署。

- **反馈循环**
    被缩短，允许更快的迭代。

#### 测试执行自动化在 DevOps 中的作用是什么？

在 DevOps 中，**[test execution automation](../T/test-execution-automation.md)** 充当 **持续集成/持续部署 (CI/CD)** 管道的关键组件。它确保对代码库所做的每项更改都会自动测试，从而快速识别缺陷并更快地向开发人员提供反馈。这种做法对于保持 DevOps 的高发布速度特征至关重要。
  每当提交新代码时，都会触发自动化测试作为 **CI 管道**的一部分。这可确保新更改不会破坏现有功能 (**[regression testing](../R/regression-testing.md)**) 并且应用程序满足指定的要求 (**[acceptance testing](../A/acceptance-testing.md)**)。通过自动化这些测试，团队可以减少手动工作，最大限度地降低人为错误的风险，并实现更频繁的[test executions](../T/test-execution.md)。
  在**CD 管道**中，自动化测试会在将应用程序部署到生产之前验证其稳定性和可靠性。这种把关功能对于确保只发布高质量的代码至关重要，从而保持生产环境的完整性。
  此外，[test execution automation](../T/test-execution-automation.md) 支持**基础设施即代码 (IaC)** 实践，允许自动化测试在通过代码配置和管理的环境中运行。这符合自动化整个软件交付流程的 DevOps 理念。
  总之，[test execution automation](../T/test-execution-automation.md) 是 DevOps 不可或缺的一部分，通过确保每个集成都经过验证、每个版本都是可靠的以及在整个开发生命周期中维护软件的整体质量，促进从开发到部署的无缝流程。

#### 如何将自动化测试集成到软件开发生命周期中？

将[automated testing](../A/automated-testing.md) 集成到软件开发生命周期 (SDLC) 中需要战略规划并遵守最佳实践。首先**建立明确的自动化目标**，并确保它们与总体项目目标保持一致。
  **在开发过程中尽早纳入自动化测试**，最好是在设计阶段。这种做法称为 **[shift-left testing](../S/shift-left-testing.md)**，有助于更快地发现问题，从而减少以后修复问题的成本和工作量。
  **选择适当的 [test cases](../T/test-case.md)** 进行自动化，重点关注那些重复的、需要多个数据集或对业务至关重要的项目。仅偶尔运行或更容易手动执行的测试可能不是自动化的最佳选择。
  **创建一个强大的[test environment](../T/test-environment.md)**，尽可能地反映生产情况。使用服务虚拟化和容器化来管理依赖关系并确保跨测试阶段的一致性。
  **开发模块化且可重用的[test script](../T/test-script.md)架构**以增强[maintainability](../M/maintainability.md)。采用 **[page object models](../P/page-object-model.md)** 或类似的设计模式将测试逻辑与 UI 结构分开，使应用程序更改时更容易更新。
  **将自动化测试与 CI/CD 管道集成**，以便在每次提交或构建时运行它们。这可确保立即反馈代码更改的影响。
  **持续监控和分析测试结果**，以识别 [flaky tests](../F/flaky-test.md) 或故障率增加的区域。使用此数据来完善您的[test suite](../T/test-suite.md) 并提高可靠性。
  **定期审查和重构**您的[test automation](../T/test-automation.md) 代码库，以保持其干净、高效并与应用程序更改保持同步。这包括删除过时的测试和更新现有的测试以反映新功能或要求。
  通过遵循这些步骤，[automated testing](../A/automated-testing.md) 成为 SDLC 的无缝且高效的部分，提供快速反馈并确保高质量的软件交付。

#### 什么是持续测试以及它与测试执行自动化有何关系？

持续测试 (CT) 是一个过程，其中自动化测试作为软件交付管道的一部分运行，以获得与候选软件发布相关的业务风险的即时反馈。 CT 是**持续集成** (CI) 和**持续部署** (CD) 不可或缺的一部分，可确保每次对代码库进行更改时自动进行测试。
  CT 与 [test execution automation](../T/test-execution-automation.md) 相关，是其核心推动者。自动化测试旨在按需触发或根据 CI/CD 管道中的特定事件触发，例如新的提交或合并的拉取请求。这可确保测试频繁且一致地运行，这对于 CT 至关重要。
  在实践中，CT 涉及：

- **自动化执行**
    单元、集成、系统和验收测试。

- **集成[test suites](../T/test-suite.md)**
    进入 CI/CD 管道以在各个阶段运行测试。

- **监控和分析**
    测试结果可以及早发现并解决问题。

- **调整测试**
    根据代码更改和新功能保持相关性。
  例如，在 CI/CD 管道中，对主分支的提交可能会触发以下自动序列：

  ```
  - Build the application
  - Run unit tests
  - Deploy to a staging environment
  - Execute integration and system tests
  - Perform exploratory testing if necessary
  - Deploy to production if all tests pass
  ```CT 的目标是在整个开发生命周期中提供持续反馈和[quality assurance](../Q/quality-assurance.md)，减少[manual testing](../M/manual-testing.md) 所需的时间和精力并提高交付速度。

- **自动化执行**
    单元、集成、系统和验收测试。

- **集成[test suites](../T/test-suite.md)**
    进入 CI/CD 管道以在各个阶段运行测试。

- **监控和分析**
    测试结果可以及早发现并解决问题。

- **调整测试**
    根据代码更改和新功能保持相关性。
