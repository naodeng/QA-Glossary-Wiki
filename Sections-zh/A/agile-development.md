敏捷软件开发是一种迭代方法，其中需求和解决方案由跨职能团队协作开发。它强调适应性和响应能力，而不是严格的计划。

## 关于敏捷开发的问题？

### 基础知识和重要性

#### 什么是敏捷开发以及为什么它很重要？

[Agile Development](../A/agile-development.md) 是一种**协作、迭代和增量**的软件开发方法，强调功能软件的**灵活性**、**客户满意度**和**快速交付**。它很重要，因为它允许团队适应不断变化的需求，通过持续反馈提高产品质量，并通过以可管理的增量发布来缩短上市时间。
  在 [test automation](../T/test-automation.md) 的背景下，[Agile Development](../A/agile-development.md) 至关重要，因为它将测试集成到开发过程中，确保快速识别和解决问题。自动化测试成为敏捷的关键资产，提供**快速反馈**并允许**持续集成**和**部署**。敏捷团队中的测试人员与开发人员和产品所有者密切合作，为用户故事做出贡献，并确保满足验收标准。 [Agile Testing](../A/agile-testing.md) 涉及从发现[bugs](../B/bug.md) 到预防它们的转变，这与敏捷对**质量**和**可持续性**的关注相一致。
  敏捷环境中的 [Test automation](../T/test-automation.md) 工程师必须善于设计、实施和维护自动化测试，这些测试必须**可靠**、**可维护**，并提供**快速反馈**。他们经常采用 **[test-driven development](../T/test-driven-development.md) (TDD)** 和 **行为驱动开发 ([BDD](../B/bdd.md))** 等实践来确保测试是从用户的角度编写的，并指导开发过程。
  [Agile Development](../A/agile-development.md) 在[test automation](../T/test-automation.md) 中的重要性怎么强调都不为过，因为它使团队能够保持高质量，同时适应现代软件交付的快节奏本质。

#### 敏捷开发的关键原则是什么？

[Agile development](../A/agile-development.md) 以敏捷宣言中概述的**四个关键原则**为基础：

1. **个人和交互**胜过流程和工具：敏捷优先考虑直接沟通和协作，重视团队响应变化的能力，而不是遵守严格的流程。
  2. **工作软件**胜过全面的文档：敏捷侧重于频繁交付功能性软件，而不那么强调详尽的文档。这并没有消除对文档的需求，但它表明进度的主要衡量标准是工作软件的交付。
  3. **客户协作**优于合同谈判：敏捷鼓励客户或利益相关者持续参与。敏捷团队不是仅仅依赖合同条款，而是与客户密切合作，以确保产品随着他们的需求和反馈而发展。
  4. **响应变化**而不是遵循计划：敏捷团队非常灵活，能够适应不断变化的需求，甚至在开发过程的后期也是如此。这种适应性被认为比严格遵循既定计划更有价值。
  这些原则指导敏捷团队的日常工作和决策过程，确保适应性、客户满意度和有效沟通处于开发工作的最前沿。作为经验丰富的 [test automation](../T/test-automation.md) 工程师，将这些原则整合到您的测试策略中将使您的工作与 [Agile development](../A/agile-development.md) 的总体目标保持一致，从而营造一个拥抱变化并专注于交付高质量功能软件的协作环境。

  2. **工作软件**胜过全面的文档：敏捷侧重于频繁交付功能性软件，而不那么强调详尽的文档。这并没有消除对文档的需求，但它表明进度的主要衡量标准是工作软件的交付。

#### 敏捷开发与传统软件开发方法有何不同？

[Agile development](../A/agile-development.md) 强调**迭代进展**、**协作**和**灵活性**，与通常依赖于**连续阶段**和**严格规划**的传统方法形成鲜明对比。在瀑布方法等传统模型中，每个阶段（需求、设计、实施、[verification](../V/verification.md)、维护）都必须在下一个阶段开始之前完成，从而形成线性和结构化的过程。
  另一方面，敏捷将产品分解为小的、可行的增量，允许对计划进行**频繁的重新评估**和**调整**。这种迭代周期有助于在每次发布时向客户**持续交付**价值，而不是等到最终产品完成。敏捷还鼓励**直接沟通**而不是文档，**客户协作**而不是合同谈判，以及**响应变化**而不是遵循既定计划。
  在实践中，敏捷团队以称为冲刺或[iterations](../I/iteration.md)的**短周期**工作，通常持续几周，以构建和交付功能性产品增量。他们举行**定期会议**，例如每日站立会议、冲刺计划和回顾，以同步工作并反思改进。测试从一开始就集成在一起，并通过**持续反馈**循环确保质量和相关性。
  敏捷的适应性使其特别适合具有**不确定或不断变化的需求**的项目，而当需求得到充分理解和稳定时，传统方法可能会更有效。敏捷中对**客户满意度**和**团队协作**的关注通常会带来更高质量的产品和更高效的开发流程。

### 敏捷方法论

#### 一些常见的敏捷方法有哪些？它们有何不同？

除了已经涵盖的 **[Scrum](../S/scrum.md)** 和 **看板** 之外，其他常见的敏捷方法还包括：

- **[Extreme Programming](../E/extreme-programming.md) (XP)：** 重点关注 **[test-driven development](../T/test-driven-development.md) (TDD)**、**重构** 和 **持续集成** 等技术实践。它强调客户满意度和迭代开发。 XP 鼓励在较短的开发周期内频繁发布，这可以提高生产力并引入可以采用新客户需求的检查点。
  - **功能驱动开发（FDD）：** 这种方法以设计和构建功能为中心。与[Scrum](../S/scrum.md) 不同，FDD 是模型驱动的，并且具有特定的角色，例如类所有权和功能团队。它包括创建总体模型、构建功能列表，然后按功能进行规划、设计和构建。
  - **精益软件开发：** 受到精益制造实践的启发，精益专注于通过消除浪费（任何不会为客户增加价值的事物）来为客户提供价值。它强调通过管理工作量和减少批量大小来优化工作流程并快速交付。
  - **动态系统开发方法 (DSDM)：** 这种方法以项目为中心，强调整个项目生命周期。 DSDM 集成了项目管理和产品开发最佳实践。它的特点是用户参与、团队有权做出决策、频繁交付产品以及以适合业务目的作为交付的主要标准。
  - **Crystal：** 敏捷方法论的集合，重点关注人员及其交互，而不是流程和工具。水晶方法针对不同的项目规模和关键程度进行定制，强调频繁交付、反思性改进和密切沟通。
  每种方法都有自己的实践和细微差别，但都共享协作、迭代开发和灵活变更的敏捷核心原则。

- **[Extreme Programming](../E/extreme-programming.md) (XP)：** 重点关注 **[test-driven development](../T/test-driven-development.md) (TDD)**、**重构** 和 **持续集成** 等技术实践。它强调客户满意度和迭代开发。 XP 鼓励在较短的开发周期内频繁发布，这可以提高生产力并引入可以采用新客户需求的检查点。
  - **功能驱动开发（FDD）：** 这种方法以设计和构建功能为中心。与[Scrum](../S/scrum.md) 不同，FDD 是模型驱动的，并且具有特定的角色，例如类所有权和功能团队。它包括创建总体模型、构建功能列表，然后按功能进行规划、设计和构建。
  - **精益软件开发：** 受到精益制造实践的启发，精益专注于通过消除浪费（任何不会为客户增加价值的事物）来为客户提供价值。它强调通过管理工作量和减少批量大小来优化工作流程并快速交付。
  - **动态系统开发方法 (DSDM)：** 这种方法以项目为中心，强调整个项目生命周期。 DSDM 集成了项目管理和产品开发最佳实践。它的特点是用户参与、团队有权做出决策、频繁交付产品以及以适合业务目的作为交付的主要标准。
  - **Crystal：** 敏捷方法论的集合，重点关注人员及其交互，而不是流程和工具。水晶方法针对不同的项目规模和关键程度进行定制，强调频繁交付、反思性改进和密切沟通。

#### 什么是 Scrum 以及它与敏捷开发有何关系？

[Scrum](../S/scrum.md) 是敏捷方法中的一个**框架**，它提供了管理和完成复杂项目的结构化方法，包括软件[test automation](../T/test-automation.md)。它强调**迭代进展**、**团队协作**和**灵活性**。
  在[Scrum](../S/scrum.md) 中，工作分为**冲刺**，通常持续一到四个星期，在此期间开发和测试产品待办事项中的特定项目。每个冲刺都以**冲刺计划**会议开始，以决定要完成的工作。 **每日 [Scrum](../S/scrum.md)** 或站立会议是一次简短的、有时间限制的会议，供团队同步活动并制定未来 24 小时的计划。
  **[Scrum](../S/scrum.md) 大师**促进这一过程，确保团队遵守[Scrum](../S/scrum.md) 实践并解决障碍。 **产品负责人**管理产品待办事项并确保团队交付价值。
  在每个冲刺结束时，团队都会进行**冲刺评审**，向利益相关者展示已完成的工作，并进行**冲刺回顾**，以反思冲刺并进行流程改进。
  [Scrum](../S/scrum.md) 与[test automation](../T/test-automation.md) 的相关性在于其适应性和对**持续反馈**的强调。 [Test automation](../T/test-automation.md) 工程师在[Scrum](../S/scrum.md) 框架内工作，根据冲刺目标开发、执行和完善自动化测试，确保测试与开发保持同步，并有助于交付高质量的软件。

#### 什么是看板以及它如何在敏捷开发中使用？

看板是一种可视化工作流程管理方法，使团队能够优化其工作流程。在 **[Agile Development](../A/agile-development.md)** 中，看板通过看板提供工作项及其状态的清晰可视化，从而提供帮助。该板分为代表开发过程不同阶段的列，例如“待办事项”、“进行中”和“已完成”。
  工作项通常以卡片的形式表示，从左向右全面移动，使团队能够跟踪进度并识别瓶颈。看板强调**限制进行中的工作** (WIP)，这鼓励集中注意力并减少多任务处理。通过为每个阶段设置在制品限制，团队可以平衡需求与吞吐量并改善流程。
  看板通过促进持续改进、灵活性和以客户为中心，与敏捷原则保持一致。它与[Scrum](../S/scrum.md) 的不同之处在于它没有规定有时间限制的[iterations](../I/iteration.md)；相反，它关注的是周期时间和吞吐量。团队在完成当前任务时拉动新的工作，使看板成为一种更加流畅和连续的方法。
  在[test automation](../T/test-automation.md) 中，看板对于管理[test case](../T/test-case.md) 开发、执行和维护流程特别有用。它允许根据反馈和不断变化的优先级进行快速调整，确保测试工作始终与最新的项目需求保持一致。
  以下是 Markdown 中简单看板布局的示例：

  ```
  | To Do | In Progress | Testing | Done |
  |-------|-------------|---------|------|
  | Task1 | Task2       | Task3   | Task4 |
  | Task5 |             |         |      |
  ```通过可视化测试活动，团队可以更有效地沟通并实时调整他们的测试策略，从而增强开发过程的整体敏捷性。

### 角色和职责

#### 敏捷团队中的角色是什么以及他们的职责是什么？

在**敏捷团队**中，角色通常不像传统方法那样严格，但关键职位包括：

- **开发团队**：负责在每个[iteration](../I/iteration.md) 结束时交付可能可交付的产品增量。他们密切合作，通常具有跨职能技能，以确保产品根据用户需求不断发展。
  - **业务分析师（BA）**：充当利益相关者和开发团队之间的桥梁。他们帮助将业务需求转化为用户故事和验收标准，确保团队了解业务背景。
  - **UX/UI 设计师**：专注于用户体验和界面设计。他们确保产品不仅功能齐全，而且直观且用户友好。
  - **[Quality Assurance](../Q/quality-assurance.md) (QA) 工程师**：与开发人员一起创建[test plans](../T/test-plan.md)，编写自动化测试，并通过各种测试方法确保产品质量。
  - **DevOps 工程师**：促进持续集成和部署 (CI/CD) 实践，维护支持 [automated testing](../A/automated-testing.md) 和高效发布管理的工具和基础设施。
  - **技术主管/架构师**：提供技术指导并确保架构支持产品的需求。他们指导团队进行技术决策和编码标准。
  每个角色都密切合作，通常身兼数职，以支持迭代开发和持续反馈的敏捷过程。重点是**团队合作、适应性**以及**为客户提供价值的承诺**。

- **开发团队**：负责在每个 [iteration](../I/iteration.md) 结束时交付可能可交付的产品增量。他们密切合作，通常具有跨职能技能，以确保产品根据用户需求不断发展。
  - **业务分析师（BA）**：充当利益相关者和开发团队之间的桥梁。他们帮助将业务需求转化为用户故事和验收标准，确保团队了解业务背景。
  - **UX/UI 设计师**：专注于用户体验和界面设计。他们确保产品不仅功能齐全，而且直观且用户友好。
  - **[Quality Assurance](../Q/quality-assurance.md) (QA) 工程师**：与开发人员一起创建[test plans](../T/test-plan.md)，编写自动化测试，并通过各种测试方法确保产品质量。
  - **DevOps 工程师**：促进持续集成和部署 (CI/CD) 实践，维护支持 [automated testing](../A/automated-testing.md) 和高效发布管理的工具和基础设施。
  - **技术主管/架构师**：提供技术指导并确保架构支持产品的需求。他们指导团队进行技术决策和编码标准。

#### Scrum Master 在敏捷开发中的角色是什么？

**[Scrum](../S/scrum.md) 大师**是敏捷[Scrum](../S/scrum.md) 团队中的促进者和教练，专注于使团队尽可能高效地工作。他们负责确保团队遵守[Scrum](../S/scrum.md) 实践和原则。 [Scrum](../S/scrum.md) Master 通过以下方式执行此操作：

- **消除障碍**：他们积极识别并消除可能阻碍团队进步的障碍。
  - **促进会议**：这包括每日站立会议、冲刺计划、冲刺审查和回顾。
  - **保护团队**：它们保护团队免受外部干扰和干扰，以保持专注于手头的任务。
  - **指导团队**：Scrum Master 帮助团队改进流程并有效地合作。
  - **确保协作**：他们鼓励团队内部以及与外部利益相关者的沟通和协作。
  - **支持产品负责人**：他们协助维护产品待办事项并确保其为下一个冲刺做好准备。
  - **促进持续改进**：Scrum Master 培育学习和适应的文化，鼓励团队反思他们的实践并不断改进。

### 敏捷实践

#### 什么是结对编程以及它如何融入敏捷开发？

结对编程是一种**敏捷软件开发技术**，其中两个程序员在一个工作站上一起工作。一个是**驱动程序**，编写代码，而另一个是**观察者**或**导航员**，检查输入的每一行代码。这两个程序员经常交换角色。
  在 **[Agile Development](../A/agile-development.md)** 的背景下，结对编程符合敏捷原则的 **协作** 和 **持续反馈** 方面。它鼓励实时代码审查和知识共享，这可以提高代码质量和团队成员技能。这种实践与敏捷对**团队合作**、**沟通**​​和**迭代进度**的强调相一致。
  结对编程还可以有助于**集体代码所有权**和**可持续的工作节奏**，这是敏捷环境中的关键。通过结对工作，团队成员可以避免专业知识孤岛，并确保有关系统不同部分的知识分布在整个团队中。
  对于[test automation](../T/test-automation.md) 工程师来说，在创建或完善自动化[test suites](../T/test-suite.md) 时，结对编程尤其有益。它允许对 [test cases](../T/test-case.md) 和脚本进行即时反馈，确保它们健壮、可理解和可维护。 [test automation](../T/test-automation.md) 中的结对编程可以带来更可靠、更有效的测试流程，这对于通常是敏捷方法论一部分的**持续集成**和**持续交付**实践至关重要。
  总之，结对编程通过促进协作、提高代码质量和共享知识来增强[Agile Development](../A/agile-development.md)，这些对于快速和自适应软件开发都是至关重要的。

#### 什么是测试驱动开发以及它如何在敏捷中使用？

[Test-Driven Development](../T/test-driven-development.md) (TDD) 是一种**软件开发实践**，其中[test cases](../T/test-case.md) *在*实际代码之前编写。在敏捷的背景下，TDD 支持**迭代开发**和**快速反馈循环**。
  以下是 TDD 在敏捷中的典型使用方式：

1. **编写失败的测试**：从尚不存在的新功能或特性开始测试。由于代码尚未实现，因此该测试应该会失败。

    ```
    describe('add function', () => {
      it('adds two numbers', () => {
        expect(add(1, 2)).toEqual(3);
      });
    });
    ```

2. **编写最简单的代码**：编写使测试通过所需的最少代码量。

    ```
    function add(a, b) {
      return a + b;
    }
    ```

3. **重构**：清理新代码，确保其与现有代码库良好契合。 [test suite](../T/test-suite.md) 确保重构不会破坏任何内容。
  4. **重复**：继续下一个测试。
  敏捷中的 TDD 可确保维持**代码质量**，最大限度地减少**回归**，并且代码库保持**灵活**以进行更改。它与敏捷对**可持续发展**和通过有价值的软件的**持续交付**的**客户满意度**的强调相一致。敏捷团队中的 [Test automation](../T/test-automation.md) 工程师利用 TDD 创建一套强大的自动化测试，随代码库一起发展，为频繁发布和重构提供信心。

#### 什么是持续集成以及它如何融入敏捷开发？

持续集成 (CI) 是一种**开发实践**，开发人员经常将代码更改合并到共享存储库中，通常一天多次。每个集成都会通过自动构建和 [test process](../T/test-process.md) 自动验证，从而使团队能够及早发现问题。
  在**[Agile Development](../A/agile-development.md)**的背景下，CI支持**快速反馈**和**持续改进**的原则。敏捷团队致力于**增量开发**，定期交付小块功能。 CI 完美地融入了这个模型，确保新的代码贡献不会破坏现有的功能，从而维护一个稳定的代码库，可以随时发布。
  对于[test automation](../T/test-automation.md) 工程师来说，CI 至关重要，因为它提供了一个框架，用于在集成过程中运行自动化测试。这意味着每次代码提交都会触发自动[test suite](../T/test-suite.md)，其中包括单元测试、集成测试，甚至可能包括验收测试。这些测试的即时反馈使开发人员能够快速解决问题，通常在代码编写后几分钟内解决，这符合敏捷对**适应性**和**客户满意度**的强调。
  以下是使用 Jenkins 的 CI 管道脚本的基本示例：

  ```
  pipeline {
      agent any
      stages {
          stage('Build') {
              steps {
                  // Commands to build the application
                  sh 'make'
              }
          }
          stage('Test') {
              steps {
                  // Commands to run automated tests
                  sh 'make test'
              }
          }
      }
      post {
          always {
              // Actions to take after the pipeline runs, like notifications
              mail to: 'team@example.com', subject: 'Build Finished'
          }
      }
  }
  ```通过定期集成，敏捷团队可以最大限度地减少等待发布日期合并功能分支时经常出现的集成挑战，从而保持**高质量的产品**。

### 敏捷和软件测试

#### 测试如何融入敏捷开发？

[Agile Development](../A/agile-development.md) 中的测试是完整且连续的，与敏捷的迭代本质相一致。它强调**早期**和**频繁**测试，确保质量从一开始就融入到产品中，而不是在最后进行检查。
  [Agile testing](../A/agile-testing.md) 涉及整个团队，开发人员、测试人员和业务利益相关者密切合作。测试人员从项目一开始就参与其中，参与需求讨论和设计会议，以了解用户故事和验收标准。这种早期参与有助于创建**相关且全面的[test cases](../T/test-case.md)**。
  **自动化**在[Agile testing](../A/agile-testing.md) 中发挥着至关重要的作用。为新功能以及[regression testing](../R/regression-testing.md) 创建了自动化测试。这些自动化测试经常运行，通常作为**持续集成 (CI)** 管道的一部分，提供有关应用程序运行状况的快速反馈。
  **[Test-Driven Development](../T/test-driven-development.md) (TDD)** 是在代码之前编写测试的常见做法。这确保了在开发的每一步都考虑测试，并且代码在被认为完成之前满足预定义的标准。
  在敏捷中，测试不是一个阶段，而是与开发并行的活动。当功能完成后，会对它们进行测试，任何问题都会立即得到解决，从而降低了累积缺陷和技术债务的风险。
  [Agile testing](../A/agile-testing.md) 具有适应性，[test plans](../T/test-plan.md) 和策略会随着项目的进展而不断发展。这种灵活性使测试过程能够快速响应需求或项目方向的变化。
  总之，敏捷测试是一个**协作、连续和自适应**过程，特别强调自动化以支持[Agile development](../A/agile-development.md)的快速步伐。
