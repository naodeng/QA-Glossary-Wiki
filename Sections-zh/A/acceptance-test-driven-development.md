验收测试驱动开发

(ATDD) 是一种开发方法，旨在通过将测试集成为开发过程的核心组件来减少缺陷。这确保了应用程序符合质量标准。

## 有关验收测试驱动开发的问题吗？

### 基础知识和重要性

#### 什么是验收测试驱动开发 (ATDD)？

[Acceptance Test Driven Development](../A/acceptance-test-driven-development.md) (ATDD) 是一种开发方法，具有不同观点的团队成员（开发人员、测试人员和业务客户）在编码开始之前协作编写验收测试。主要目标是为系统功能指定详细的、以客户为中心的标准，以指导开发并提供对期望结果的清晰理解。
  在 ATDD 中，验收测试以示例或场景的形式表示，通常使用“Given-When-Then”格式，从用户的角度描述系统的行为。这些测试是自动化的，并作为实时文档和回归套件。
  ATDD 促进团队成员之间更好的沟通和理解，确保功能满足业务需求。它使开发工作与客户需求保持一致，并有助于防止功能蔓延和缺陷。通过从一开始就关注客户需求，团队可以交付更有价值、更高质量的软件。
  测试人员在 ATDD 中的角色超出了传统测试范围，包括参与需求澄清并确保验收标准可测试且清晰。测试人员与开发人员和业务代表密切合作，创建和自动化验收测试。
  ATDD 的常用工具包括 Cucumber、SpecFlow 和 FitNesse，它们支持行为驱动开发 ([BDD](../B/bdd.md)) 和示例规范实践。这些工具允许用所有利益相关者都能理解的语言编写测试，从而弥合了技术和非技术团队成员之间的差距。
  有效实施 ATDD 需要转变思维方式和实践，强调前期规范、持续反馈和迭代开发。它是敏捷和精益开发方法的关键实践，有助于交付满足用户期望的高质量软件。

#### 为什么 ATDD 在软件开发中很重要？

ATDD 在软件开发中至关重要，因为它确保所有利益相关者在编码开始之前对需求有共同的理解。这种方法使开发人员、测试人员和业务代表围绕商定的验收标准保持一致，从而促进更好的沟通和协作。通过从一开始就关注客户的需求，ATDD 最大限度地减少了误解的风险，并降低了开发周期后期进行代价高昂的返工的可能性。
  在开发过程的早期纳入 ATDD 有助于在问题升级之前识别和解决问题，从而实现更高效、更精简的工作流程。它鼓励使用简单的语言进行行为规范，使所有相关方都能理解测试。这种清晰度有助于预防缺陷，而不是在开发后检测缺陷，与传统测试方法相比，这是一种更主动的方法。
  ATDD 还促进持续反馈，从而允许在整个开发生命周期中进行迭代改进。这种迭代过程有助于完善产品，以更好地满足用户期望，最终产生与业务目标紧密结合的更高质量的软件。
  此外，ATDD 对自动化的强调支持 [regression testing](../R/regression-testing.md) 并实现可持续的测试流程，无需显着增加工作量即可处理更改。这个[automated testing](../A/automated-testing.md) 框架对于在快节奏的开发环境中保持高水平的质量至关重要，特别是在扩展到更大的项目时。
  总之，ATDD 很重要，因为它促进共识、减少返工、确保与业务目标保持一致，并支持可持续的高质量开发流程。

#### ATDD 与传统测试方法有何不同？

ATDD 或 [Acceptance Test Driven Development](../A/acceptance-test-driven-development.md) 与传统测试方法的不同之处主要在于其**协作方法**和**计时**。传统测试通常发生在开发阶段之后，测试人员根据已实现的功能创建和执行测试。相比之下，ATDD 涉及多个利益相关者，包括开发人员、测试人员和业务代表，他们在编写任何代码之前定义验收标准并创建验收测试。
  ATDD 中的这种前期协作确保各方对需求和“完成”的定义有共同的理解。它将重点从开发后查找[bugs](../B/bug.md) 转移到通过尽早澄清期望来防止[bugs](../B/bug.md)。此外，ATDD 鼓励**行为驱动的开发** ([BDD](../B/bdd.md))，其中测试以所有利益相关者都可以理解的语言编写，通常使用“Given-When-Then”格式。
  虽然传统的测试方法可能严重依赖 [manual testing](../M/manual-testing.md) 或在事后创建自动化测试，但 ATDD 从一开始就集成了 **[test automation](../T/test-automation.md)**。验收测试是自动化的，并成为**回归套件**的一部分，提供有关新更改是否符合商定标准的即时反馈。
  总之，ATDD 的主动、协作方法与传统的反应性测试方法形成鲜明对比，强调预防而非检测，并促进整个团队共同承担质量责任。

#### 使用 ATDD 的主要好处是什么？

ATDD 提供了增强软件开发过程的几个关键优势：

- **增强协作**：通过在开发周期的早期让各个利益相关者（开发人员、测试人员、业务分析师）参与其中，ATDD 可以促进更好的理解和沟通。
  - **明确的需求**：验收测试作为具体的需求，减少歧义并确保软件满足业务需求。
  - **早期缺陷检测**：当预先定义验收标准时，可以更早地发现问题，从而减少以后修复错误的成本和工作量。
  - **客户满意度**：注重满足验收标准，确保最终产品符合客户期望。
  - **回归安全**：自动化验收测试提供了一个安全网，使重构和改进代码更加安全，而不会破坏现有功能。
  - **持续反馈**：定期执行验收测试可以持续洞察产品状态，以便及时调整。
  - **简化开发**：明确的验收标准指导开发工作，防止功能蔓延和过度设计。
  实施 ATDD 可以带来更加高效、协作和注重质量的开发生命周期，最终交付能够更好地满足用户需求并经受时间考验的软件。

#### ATDD 如何提高软件产品的质量？

ATDD 通过确保精确满足 **[functional requirements](../F/functional-requirements.md)** 以及产品的行为符合利益相关者的预期来增强 [software quality](../S/software-quality.md)。通过从一开始就关注客户需求，ATDD 促进创建**清晰且可执行的规范**。这些规范指导开发和测试，减少由于误解或不完整的需求而出现缺陷的可能性。
  验收测试是在编写代码之前定义的，这意味着开发人员有明确的目标。这种“测试优先”方法有助于防止功能蔓延，并确保代码库仅包含通过测试所需的内容，从而形成更干净、更易于维护的代码库。
  此外，ATDD 鼓励开发人员、测试人员和业务利益相关者之间的协作。这种**跨职能沟通**有助于在开发过程的早期识别和解决歧义，从而显着提高产品的质量。
  执行验收测试的持续反馈可以**及早发现问题**，这通常比开发周期后期发现的[bugs](../B/bug.md) 修复成本更低。此外，这套验收测试成为可用于 **[regression testing](../R/regression-testing.md)** 的动态文档，确保新的更改不会破坏现有功能。
  总之，ATDD 通过澄清需求、促进协作和提供持续反馈来为 [software quality](../S/software-quality.md) 做出贡献，所有这些都有助于构建与业务需求和用户期望紧密结合的产品。

### 流程和技术

#### ATDD 涉及哪些关键步骤？

ATDD 涉及的关键步骤是：

1. **合作**
    开发人员、测试人员和业务利益相关者之间共同定义验收标准。

2. **创建验收测试**
    在开发开始之前，根据商定的标准。

3. **开发**
    以验收测试为指导，描述功能或用户故事。

4. **持续集成**
    确保根据验收测试自动测试代码更改。

5. **细化**
    必要时进行验收测试，以解决需求或理解的变化。

6. **[Test Execution](../T/test-execution.md)**
    验证软件是否符合商定的验收标准。

7. **审查和反馈**
    利益相关者确认验收测试涵盖了所需的功能和行为。

8. **[Iteration](../I/iteration.md)**
    根据需要执行这些步骤，直到功能满足验收标准。
  验收测试通常是自动化的，以方便频繁执行和[regression testing](../R/regression-testing.md)。测试以所有相关方都能理解的语言编写，通常使用行为驱动开发 ([BDD](../B/bdd.md)) 框架，例如 Cucumber 或 SpecFlow。这确保了测试既可以作为规范又可以作为验证。

  ```有效的 ATDD 需要强大的协作文化、清晰的沟通以及所有团队成员对质量的承诺。
#### ATDD 中如何创建验收测试？

ATDD 中的验收测试是通过团队成员（包括开发人员、测试人员和业务利益相关者）之间的**协作**创建的。该过程首先定义**用户故事**，从用户的角度描述所需的功能。每个用户故事都包含**验收标准**，作为验收测试的基础。
  团队讨论验收标准以确保对需求有共同的理解。这种讨论通常涉及**示例映射**或**示例规范**，其中使用具体示例来阐明期望并涵盖不同的场景。
  一旦标准达成一致，**可执行的规范**就会被写入。这些通常被构造为 **Given-When-Then** 语句，可以直接转换为自动化测试。例如：

  ```
  Given the user is logged in
  When they attempt to place an order
  Then the order should be processed
  ```然后使用 **Cucumber** 或 **SpecFlow** 等 ATDD 框架自动化这些规范，该框架允许使用非技术利益相关者可以访问的特定于领域的语言编写测试。自动化代码是用与框架兼容的语言编写的，例如用于 Cucumber 的 Java 或用于 SpecFlow 的 C#。

  ```
  @Given("^the user is logged in$")
  public void the_user_is_logged_in() {
      // Code to ensure user is logged in
  }
  @When("^they attempt to place an order$")
  public void they_attempt_to_place_an_order() {
      // Code to simulate order placement
  }
  @Then("^the order should be processed$")
  public void the_order_should_be_processed() {
      // Assertions to verify order processing
  }
  ```在整个开发过程中不断执行验收测试，以验证软件是否满足商定的标准。这确保了在开发新功能时考虑到用户的需求，并尽早发现回归问题。

#### ATDD 中常用哪些技术？

在 ATDD 中，采用了多种技术来确保满足验收标准并且软件按预期运行：

- **行为驱动开发 ([BDD](../B/bdd.md))**：此技术涉及以自然语言风格编写描述应用程序行为的测试。 Cucumber 或 SpecFlow 等工具通常用于促进 [BDD](../B/bdd.md)。
  - **示例规范**：协作定义说明特定行为或要求的示例。然后将这些示例用作验收测试的基础。
  - **示例映射**：一种研讨会技术，团队成员使用卡片来表示用户故事（黄色）、规则（蓝色）、示例（绿色）和问题（红色）。这有助于理解故事并创建验收测试。
  - **可执行规范**：以可以直接针对代码执行的方式编写验收测试。这通常涉及特定领域语言（DSL），以所有利益相关者都可以理解的方式表达测试。
  - **测试优先开发**：在实际实施开始之前编写验收测试，确保开发的重点是通过测试。
  - **协作工具**：使用促进业务利益相关者、开发人员和测试人员之间协作的工具，例如共享存储库或协作平台，例如带有 Xray 或 TestRail 的 [JIRA](../J/jira.md)。
  - **持续集成 (CI)**：作为 CI 管道的一部分自动运行验收测试，以获得有关所做更改的即时反馈。
  - **测试工件的版本控制**：将验收测试与代码库一起存储在版本控制系统中，以保持 [test cases](../T/test-case.md) 和应用程序代码之间的同步。
  这些技术有助于定义明确的验收标准，促进团队成员之间的协作，并确保软件在被认为完成之前满足业务需求。

- **行为驱动开发 ([BDD](../B/bdd.md))**：此技术涉及以自然语言风格编写描述应用程序行为的测试。 Cucumber 或 SpecFlow 等工具通常用于促进[BDD](../B/bdd.md)。

  - **测试优先开发**：在实际实施开始之前编写验收测试，确保开发的重点是通过测试。
  - **协作工具**：使用促进业务利益相关者、开发人员和测试人员之间协作的工具，例如共享存储库或协作平台，例如带有 Xray 或 TestRail 的[JIRA](../J/jira.md)。
  - **持续集成 (CI)**：作为 CI 管道的一部分自动运行验收测试，以获得有关所做更改的即时反馈。

#### ATDD 如何集成到软件开发生命周期中？

ATDD 通过从一开始就将开发活动与指定的验收标准相结合来集成到软件开发生命周期 (SDLC) 中。在**初始阶段**，利益相关者协作定义和理解需求，然后将其转化为验收测试。这些测试代表了软件必须表现出的**行为**才能被视为完整。
  在**规划和设计**期间，将对验收测试进行审查和完善，确保它们清晰且可测试。开发人员、测试人员和业务代表保持**持续沟通**以澄清任何歧义。
  随着开发的开始，**验收测试指导编码**。开发人员编写足够的代码来通过这些测试，确保功能满足商定的标准。这种做法通常称为“测试优先开发”，可促进“渐进式进展”并有助于及早发现问题。
  在**测试阶段**，经常执行自动验收测试，提供有关软件功能的**即时反馈**。这允许快速调整并有助于保持发展的**稳定步伐**。
  在发布之前，ATDD 确保产品满足业务要求，因为所有功能都是根据预定义的验收测试开发的。代码的**持续集成**和验收测试的**定期执行**有助于维护为部署做好准备的**稳定的构建**。
  部署后，这些验收测试成为**回归套件**的一部分，防止未来可能破坏现有功能的更改。 ATDD 与 SDLC 的集成支持**可持续的、注重质量的开发流程**。

#### 测试人员在 ATDD 中的角色是什么？

在 ATDD 中，测试人员的角色是多方面的，重点是协作、规范和验证。测试人员与**开发人员**、**业务分析师**和**利益相关者**密切合作，以澄清需求并确保明确定义验收标准。他们有助于**创建用户故事**和**验收测试**，确保这些测试准确反映业务需求并且可以自动化。
  在开发过程中，测试人员参与验收测试的**持续改进**，通常与开发人员合作创建和维护[test automation](../T/test-automation.md)。他们确保测试是**可执行的规范**，指导开发并根据预期结果提供有关软件行为的即时反馈。
  测试人员还在**维护[test suite](../T/test-suite.md)**方面发挥着关键作用，确保它随着代码库的发展保持**可靠**和**高效**。他们还可能负责**[test data](../T/test-data.md) 管理**并设置必要的**[test environments](../T/test-environment.md)**。
  在 ATDD 周期中，测试人员帮助促进**三场 Amigos 会议**，在会上讨论功能实现，并且他们积极参与**[iteration](../I/iteration.md) 规划**和**回顾**，以不断改进流程。
  最终，ATDD 的测试人员确保团队交付的产品不仅满足技术要求，而且满足业务需求并为最终用户提供价值。它们是推动 ATDD 固有的**质量第一方法**的关键。

#### 这些工具如何支持 ATDD 流程？

[Test automation](../T/test-automation.md) 工具通过启用以 **行为驱动** 格式编写的验收测试的 **执行** 来促进 **ATDD 过程**。这些工具通常与 **Cucumber**、**SpecFlow** 或 **FitNesse** 等框架集成，这些框架允许使用 **业务可读** 语言定义测试，例如 [Gherkin](../G/gherkin.md)。
  通过使用这些工具，团队可以自动执行验收标准的**验证**，确保软件符合商定的规范。这种自动化通过允许在代码签入时自动运行测试来支持**持续集成**（CI）实践，为开发人员提供**即时反馈**。
  此外，[test automation](../T/test-automation.md) 工具通过维护一套测试来支持**重构**，这些测试可以运行以验证更改没有破坏现有功能。这在 ATDD 中至关重要，重点是在整个开发过程中满足验收标准。
  此外，这些工具通常带有**报告功能**，可以更轻松地向所有利益相关者传达测试状态。这种透明度有助于保持对项目进度和质量的**共同理解**。
  例如，典型的 ATDD 工具链可能如下所示：

  ```
  Feature: User login
    Scenario: Valid user login
      Given I am on the login page
      When I enter valid credentials
      Then I should be redirected to the dashboard

  ```
#### 自动化在 ATDD 中的作用是什么？

在 ATDD 中，自动化在一致且高效地**验证**和**回归**验收标准方面发挥着关键作用。自动化测试源自团队商定的验收标准，确保软件符合业务需求。 ATDD 中的自动化：

- **促进持续测试**
    通过将测试集成到 CI/CD 管道中，可以立即反馈更改。

- **确保重复性**
    验收测试的一致性，减少测试执行中的人为错误和可变性。

- **增加[test coverage](../T/test-coverage.md)**
    通过允许在更短的时间内执行更多的测试。

- **增强协作**
    通过提供所有团队成员都可以理解并用于验证系统的清晰、可执行的规范。

- **缩短反馈循环**
    ，因为可以频繁运行自动化验收测试，从而可以快速了解软件的状态。

- **支持重构**
    通过提供一个安全网来确保新的更改不会破坏现有的功能。
  自动化验收测试成为始终最新的系统行为的活文档。它们通常是用**特定于领域的语言 (DSL)** 编写的，使非技术利益相关者也可以访问它们。 Cucumber、SpecFlow 或 FitNesse 等工具通常用于促进此过程。
  ```

#### ATDD 如何在敏捷开发环境中实现？

在敏捷环境中实施 ATDD 需要开发人员、测试人员和业务利益相关者之间的协作，以在开发开始之前定义验收标准。使用**用户故事**来捕获需求并定义反映利益相关者期望的行为的**验收测试**。
  从**计划会议**开始讨论用户故事并创建验收测试。这确保了对该功能及其预期结果的共同理解。使用 Cucumber 或 SpecFlow 等**行为驱动开发 ([BDD](../B/bdd.md)) 工具**，以所有相关方都能理解的语言编写验收测试。
  在**开发**期间，程序员和测试人员协同工作，测试人员专注于自动化验收测试。这些测试被集成到**持续集成（CI）管道**中，确保它们频繁运行。
  开发完成后，执行验收测试。如果通过，则该功能符合商定的标准。如果没有，开发人员会进行必要的更改。此循环一直持续到该功能通过所有验收测试。
  通过定期与团队（包括利益相关者）一起审查测试及其结果，纳入**反馈循环**。这确保了验收标准与业务目标保持一致，并且任何误解都能得到及时解决。
  请记住**重构**代码和测试以保持简单性和可读性。这种做法有助于保持自动化套件的可维护性和可扩展性。
  最后，确保团队**跨职能**，成员能够为开发和测试做出贡献。这种方法培养了一种质量文化和对最终产品的共同责任。

### 挑战和解决方案

#### ATDD 中遇到的常见挑战有哪些？

ATDD 的常见挑战包括：

- **协作困难**：确保开发人员、测试人员和业务利益相关者之间的有效沟通可能具有挑战性。误解可能导致不正确的测试标准。
  - **编写明确的验收标准**：制定明确的、可测试的验收标准需要技能和经验。写得不好的标准可能会导致测试无效。
  - **维护[test suites](../T/test-suite.md)** ：随着应用程序的发展，保持验收测试最新可能非常耗时。
  - **[Test data](../T/test-data.md) 管理**：生成和管理验收测试所需的数据可能很复杂，尤其是在处理多个环境时。
  - **平衡覆盖率与速度**：实现足够的测试覆盖率，同时保持测试套件足够快以进行持续集成可能很困难。
  - **[Flaky tests](../F/flaky-test.md)** ：间歇性地通过和失败的测试可能会削弱对测试套件的信心并浪费调查时间。
  - **与现有流程集成**：将 ATDD 引入已建立的工作流程可以克服阻力，并且需要仔细的变更管理。
  - **工具兼容性**：确保所选工具与技术堆栈良好集成并支持 ATDD 方法可能是一个障碍。
  - **技能组**：团队成员可能需要培训才能有效参与 ATDD，包括编写验收测试并将其自动化。
  克服这些挑战通常涉及改善沟通、投资培训、完善流程以及选择符合团队需求和技术堆栈的适当工具。

#### 如何克服这些挑战？

克服 ATDD 的挑战需要采取战略方法：

- **协作**：培养开发人员、测试人员和业务利益相关者之间的协作文化，以确保对需求的共同理解。
  - **持续学习**：鼓励团队成员不断学习和适应新工具和技术，以跟上 ATDD 最佳实践。
  - **渐进式改进**：从小处开始，逐步改进 ATDD 流程，而不是尝试大规模检修。
  - **工具熟练程度**：投入时间掌握 ATDD 工具，以充分发挥其潜力并将其无缝集成到开发工作流程中。
  - **反馈循环**：实施简短的反馈循环以快速识别和解决问题，从而提高验收测试的质量。
  - **重构**：定期重构测试以保持清晰度并减少维护开销。
  - **模块化设计**：将测试设计为模块化且可重用，以最大限度地减少重复并简化更新。
  - **清晰的文档**：维护清晰的测试文档，以确保所有团队成员都能理解和维护它们。
  - **资源分配**：分配足够的资源，包括时间和人员，以保持可持续的步伐并避免倦怠。
  - **指标**：使用指标来跟踪进度并确定需要改进的领域，但避免使用会激励错误行为的指标。
  - **风险管理**：根据风险和业务价值确定测试的优先级，以确保关键功能得到彻底测试。

#### ATDD 如何针对大型项目进行扩展？

扩展大型项目的 ATDD 需要战略规划和高效的工具。首先**构建验收测试**以反映项目的模块化架构。这允许跨不同团队并行开发和测试。利用**版本控制**来管理测试工件并确保跨团队的同步。
  利用**[test data](../T/test-data.md) 管理**提供一致且隔离的[test environments](../T/test-environment.md)，避免共享数据可能出现的冲突和依赖性。实施**持续集成 (CI)** 以针对新代码提交自动运行验收测试，提供有关集成状态的即时反馈。
  **分布式[test execution](../T/test-execution.md)**对于处理增加的测试负载至关重要。使用支持跨多台机器或容器并行运行测试的工具。这减少了反馈循环并确保更快的周转时间。
  **协作工具**对于维持开发人员、测试人员和利益相关者之间的沟通至关重要。这些工具应支持从需求到测试和代码库的可追溯性，确保各方在验收标准上保持一致。
  **指标和报告**应进行定制，以提供对大规模进展和质量的见解。自动化仪表板可以跟踪[test coverage](../T/test-coverage.md)、通过/失败率以及一段时间内的趋势，有助于及早识别关注领域。
  最后，尽可能模块化和重用测试组件。测试步骤或领域特定语言 (DSL) 定义的共享库可以减少重复和维护开销。
  通过关注这些策略，ATDD 可以有效地扩展以适应大型项目的复杂性。

#### 如何衡量 ATDD 的有效性？

ATDD的有效性可以通过几个关键指标来衡量：

- **减少缺陷率**：跟踪发布后发现的缺陷数量。较低的数字表明 ATDD 有助于更早地发现并解决问题。
  - **改进[test coverage](../T/test-coverage.md)**：使用覆盖工具确保验收测试覆盖代码库和用户故事的很大一部分。
  - **周期时间**：监控从功能构思到交付的时间。 ATDD 应有助于简化流程，从而缩短周期时间。
  - **反馈循环持续时间**：衡量从利益相关者那里收到反馈需要多长时间。 ATDD 旨在缩短这个循环，从而实现更快的调整。
  - **团队协作**：评估开发人员、测试人员和业务利益相关者之间的协作水平。有效的 ATDD 实践应该加强沟通和理解。
  - **回归[test suite](../T/test-suite.md)执行时间**：跟踪运行回归套件所需的时间。 ATDD 应该会带来更高效、更有针对性的测试，从而减少执行时间。
  - **验收测试的通过/失败率**：记录首次运行的验收测试的通过率。通过率高表明团队对需求有很好的理解。
  - **客户满意度**：调查利益相关者和最终用户，以评估他们对所提供功能的满意度。较高的满意度表明 ATDD 实施成功。
